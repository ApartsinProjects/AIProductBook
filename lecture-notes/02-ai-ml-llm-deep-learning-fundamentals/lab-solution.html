Exercise 1 Solution

Inspect Tokenization And Context Cost

Supporting lecture sections: Tokens And Embeddings, Transformers

Original Assignment

Pick five short strings that look similar in visible length but differ in structure, for example: a plain sentence, a URL, a JSON blob, code, and a sentence with punctuation-heavy formatting.

  1. Estimate which ones you think will produce more tokens and why.
  2. Use a tokenizer tool or reason carefully from known tokenization behavior.
  3. Explain how token count affects cost, latency, and context fit in a production system.
  4. State one practical implication for prompt design or chunking.

Expected Deliverable

Create a one-page note with your five examples, tokenization observations, and one engineering takeaway.

Worked Example

StringLikely token behaviorReason
`Reset my password please.`Relatively compactCommon words are often tokenized efficiently
`https://app.example.com/reset?token=abc123`Often higher token count than expectedURLs mix punctuation, symbols, and uncommon fragments
`{"user":"ana","action":"reset"}`Can be token-heavy for its sizeJSON punctuation and quoting patterns often fragment
`if (!user) return 401;`Can tokenize differently from plain languageCode uses symbols and syntax-specific fragments
`Reset!!! Password??? Now...`Often less efficient than plain proseHeavy punctuation can increase token fragmentation

Engineering Takeaway

Visible character count is not enough. Structured or punctuation-heavy text can consume more tokens than students expect, which directly affects prompt cost and context budgets.

Why This Matters Later

This becomes crucial in prompt compression, chunking, RAG context assembly, and latency optimization.

Exercise 2 Solution

Reason About Embedding Similarity

Supporting lecture sections: Tokens And Embeddings, Why Neural Networks Matter

Original Assignment

Build a small set of 8-10 support-style sentences. Include pairs that are semantically similar but not string-similar, and pairs that are string-similar but semantically different.

  1. Group the sentences into clusters you expect embeddings to place near each other.
  2. Explain why semantic similarity can differ from keyword overlap.
  3. Identify one case where embeddings may still fail without better context or metadata.

Expected Deliverable

Create a small similarity table or cluster diagram and a short written explanation.

Worked Example Clusters

SentenceExpected clusterWhy
`I can't log in to my account`AuthenticationLogin/access problem
`reset my password`AuthenticationClosely related intent though different wording
`my invoice total looks wrong`BillingPayment/accounting issue
`why was I charged twice?`BillingSame business intent, different wording
`account page won't load`May be ambiguousCould be auth, UI, or outage depending on context

Key Concept

Embeddings help when meaning matters more than literal overlap.

Where Embeddings Still Fail

Ambiguous short texts may need metadata, user history, or product context.

Exercise 3 Solution

Choose A Threshold And Defend It

Supporting lecture sections: Metrics And Thresholds, Capabilities, Limits, And Failures

Original Assignment

Imagine you are building an inbound-support triage system that predicts whether a ticket should be escalated to a senior specialist. You are given these model scores for ten cases:

0.94, 0.89, 0.81, 0.77, 0.72, 0.68, 0.61, 0.54, 0.40, 0.22

Assume that the truly escalation-worthy cases are the first, second, fourth, fifth, and seventh items in the list.

  1. Evaluate two thresholds of your choice, for example `0.80` and `0.60`.
  2. Compute the resulting true positives, false positives, false negatives, precision, and recall.
  3. Choose a threshold and defend it in product terms, not just mathematical terms.
  4. State whether calibration or human review policy should be part of the design.

Expected Deliverable

Create a short threshold-analysis memo with your calculations and recommendation.

Worked Calculation

ThresholdPredicted positivesTPFPFNPrecisionRecall
0.801, 2, 32132/3 = 0.672/5 = 0.40
0.601, 2, 3, 4, 5, 6, 75205/7 = 0.715/5 = 1.00

Recommendation

If the system is only routing cases into a human review queue, a threshold near `0.60` may be preferable because missing escalation-worthy tickets is costly.

Why Calibration Still Matters

Confidence scores should not be treated as perfectly trustworthy.

Exercise 4 Solution

Design A Fundamentals-Aware AI Feature

Supporting lecture sections: The Large-Picture Map For This Module, Metrics And Thresholds, Module 1 Prototype Versus Production

Original Assignment

Design a narrow AI feature for one business workflow. Keep the scope small enough that you can clearly justify the model choice, threshold policy, and likely failure modes.

  • Describe the user problem in 2-3 sentences.
  • State which model behaviors matter most: classification, extraction, ranking, generation, or semantic retrieval.
  • List the core fundamentals that matter to your design: tokens, embeddings, thresholds, context window, training assumptions, or calibration.
  • State one thing the system should definitely not do automatically.

Expected Deliverable

Turn in a short design note with a problem statement, design choices, threshold stance, and failure/risk notes.

Worked Example: Escalation-Recommendation Assistant

Problem:
Support leads want help identifying tickets that probably need escalation.

Model behaviors:
- classification
- semantic retrieval of similar past cases

Fundamentals that matter:
- embeddings for similar-case retrieval
- threshold choice for escalation recommendation
- calibration of confidence scores
- token cost when attaching long ticket histories

What the system should not do automatically:
- it should not escalate premium or legal-risk tickets without human review

Why This Is Strong

  • The scope is narrow and operationally clear.
  • The design uses fundamentals, not just buzzwords.
  • The review boundary is explicit.

Common Weak Answer

“Use an LLM to handle support better” is too vague.
