Module 1 Lab

Lab: Map, Critique, and Design a Real AI System

This lab turns the lecture into concrete architecture thinking. Students analyze real products, map the layers of a model-powered system, and design a first-pass AI application with explicit choices about autonomy, humans, and production risk.

Open companion lecture notes

Overview

Lab Objectives

Objective 1

Identify the major components inside real AI products.

Objective 2

Practice distinguishing model behavior from application behavior.

Objective 3

Design a minimal but realistic model-powered system with clear ownership and failure analysis.

Recommended time: 90-120 minutes. This lab works well individually or in pairs.

Preparation

Before You Start

  • Read What Problem Does AI Engineering Solve?
  • Review What Kinds Of Things Do AI Engineers Build?
  • Study What Is Inside A Model-Powered Application?
  • Skim What Usually Goes Wrong?
Exercise 1

Map Three Real AI Products

Pick three AI products from different categories, for example a coding assistant, a support copilot, and a document Q&A tool. For each one, answer the questions below.

  1. What is the user trying to accomplish?
  2. Which product type does it match from the lecture taxonomy?
  3. Where is the model probably being used?
  4. What non-model components must exist around it?
  5. What would likely fail first if the system moved from demo to production?

Deliverable

Create a short comparison table with one row per product and columns for user goal, AI pattern, system components, and likely failure mode.

Exercise 2

Draw the System Anatomy

Choose one product from Exercise 1 and draw a first-pass architecture using the component pattern from the system anatomy section.

  • User interface or upstream system
  • Application layer
  • Prompt and context assembly
  • Retrieval or tools if needed
  • State or storage
  • Observability
  • Validation or human approval where appropriate

Suggested format

User -> App API -> Orchestrator -> Model
                  |              |
                  v              v
              Storage        Retrieval / Tools
                  |
                  v
           Logs / Metrics / Review Queue

Deliverable

Turn in a one-page architecture sketch with short notes explaining what each box is responsible for.

Exercise 3

Predict Failure Modes

Use the framework from the failure-modes section to predict where your chosen system will break.

  1. Name one likely model failure.
  2. Name one likely context or retrieval failure.
  3. Name one likely workflow or UX failure.
  4. Name one likely operational failure.
  5. For each failure, suggest one mitigation.

Deliverable

Create a risk table with columns for failure, layer, user impact, and mitigation.

Exercise 4

Design a Minimal AI Application

Now create your own simple AI product idea. Keep it narrow. The goal is not novelty. The goal is to make a clean architecture decision and justify it.

  • Write the user problem in 2-3 sentences.
  • Choose the product pattern: copilot, knowledge system, workflow automation, decision support, agent, or multimodal app.
  • Decide the autonomy level using the logic from the autonomy ladder.
  • List the minimum components needed to make it useful.
  • State what makes it a prototype versus what would be needed for production.

Assessment

Rubric

Strong Submission

  • Correctly distinguishes model, app, data, and operational layers
  • Makes a sensible autonomy choice
  • Identifies realistic risks and mitigations
  • Shows prototype versus production thinking clearly

Weak Submission

  • Treats the model as the whole system
  • Assumes production means only better prompts
  • Ignores human review, observability, or failure handling
  • Cannot explain why AI belongs in the workflow
Wrap-Up

What This Lab Should Teach

If students finish this lab well, they should stop seeing AI products as mysterious black boxes. They should instead see them as systems with understandable parts, tradeoffs, and ownership boundaries.
