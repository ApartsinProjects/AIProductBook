Exercise 1 Solution

Map Three Real AI Products

Supporting lecture sections: Big Picture, Product Types, System Anatomy

Original Assignment

Pick three AI products from different categories, for example a coding assistant, a support copilot, and a document Q&A tool. For each one, answer the questions below.

  1. What is the user trying to accomplish?
  2. Which product type does it match from the lecture taxonomy?
  3. Where is the model probably being used?
  4. What non-model components must exist around it?
  5. What would likely fail first if the system moved from demo to production?

Expected Deliverable

Create a short comparison table with one row per product and columns for user goal, AI pattern, system components, and likely failure mode.

Worked Example Table

Invoice intake toolExtract fields from documentsExtraction workflowFile ingest, OCR, parsing, validation rules, review queue, audit logDocument variability breaks extraction on edge cases

Why This Is Strong

  • It identifies both model and non-model components.
  • It maps products to the lecture taxonomy instead of treating all AI apps as chatbots.
  • It predicts realistic failure points.

Common Weak Answer

“This product uses GPT to answer questions” is too shallow because it ignores context, retrieval, validation, UI, permissions, and operations.

Exercise 2 Solution

Draw The System Anatomy

Supporting lecture sections: System Anatomy, Roles

Original Assignment

Choose one product from Exercise 1 and draw a first-pass architecture using the component pattern from the lecture.

  • User interface or upstream system
  • Application layer
  • Prompt and context assembly
  • Retrieval or tools if needed
  • State or storage
  • Observability
  • Validation or human approval where appropriate

Expected Deliverable

Turn in a one-page architecture sketch with short notes explaining what each box is responsible for.

Worked Example: Support Copilot Architecture

Agent UI
  -> Support API
    -> Auth + permissions
    -> Context builder
      -> Retrieve product docs
      -> Retrieve customer account context
    -> Model call
    -> Output validator
    -> Citation formatter
    -> Trace + metrics logger
    -> Return draft reply

Fallback path:
  low confidence or validation failure
    -> human review queue

Why These Boxes Matter

  • The API owns business logic.
  • The context builder decides what the model sees.
  • Validation protects the user from brittle outputs.
  • Logging and traces make debugging possible later.

What Students Should Notice

The architecture is still understandable as normal software. The model is important, but it is not the whole application.

Exercise 3 Solution

Predict Failure Modes

Supporting lecture sections: Failure Modes, Prototype vs Production

Original Assignment

Use the lecture failure-mode framework to predict where your chosen system will break.

  1. Name one likely model failure.
  2. Name one likely context or retrieval failure.
  3. Name one likely workflow or UX failure.
  4. Name one likely operational failure.
  5. For each failure, suggest one mitigation.

Exercise 4 Solution

Design A Minimal AI Application

Supporting lecture sections: Big Picture, Product Types, Prototype vs Production, Code Examples

Original Assignment

Design your own simple AI product idea. Keep it narrow. The goal is not novelty. The goal is to make a clean architecture decision and justify it.

  • Write the user problem in 2-3 sentences.
  • Choose the product pattern: copilot, knowledge system, workflow automation, decision support, agent, or multimodal app.
  • Decide the autonomy level using the logic from the lecture autonomy ladder.
  • List the minimum components needed to make it useful.
  • State what makes it a prototype versus what would be needed for production.

Expected Deliverable

Turn in a short design note using the provided template or an equivalent structure.

Worked Example: Meeting Note Action-Item Extractor

Problem:
Teams lose action items buried in meeting notes and transcripts.

Pattern:
This is mainly an extraction and workflow automation system.

Why AI is useful:
Action items are phrased inconsistently and often implied rather than stated in a strict template.

System components:
1. Transcript input or note upload
2. Extraction API
3. Prompt + schema for action item extraction
4. Validator for owners and deadlines
5. Review UI for low-confidence items
6. Export to task tracker

Autonomy choice:
Act with approval. The system should draft action items, but a human should confirm before tasks are created.

Prototype scope:
Prove that the system can extract useful action items from 20 realistic meeting notes.

Production additions:
Authentication, task-system integration, audit logs, eval set, retry logic, dashboard, and error handling.

Why This Is A Good Module 1 Answer

  • It is narrow and concrete.
  • It explains why AI is needed instead of rules alone.
  • It makes an explicit autonomy decision.
  • It distinguishes prototype goals from production needs.

What Would Make It Weaker

  • Making it a fully autonomous meeting agent too early
  • Skipping review or validation
  • Claiming production readiness after only a happy-path demo
Wrap-Up

What A Strong Submission Shows

  • The student can reason in terms of systems, not just model prompts.
  • The student can identify realistic components and responsibilities.
  • The student can anticipate risk and justify autonomy choices.
  • The student understands the difference between a promising demo and a production-ready system.

Complete Assignment

Full Lab Challenge At A Glance

The complete Module 1 challenge is to:

  1. Analyze three real AI products and classify their system type.
  2. Draw the architecture of one product using the lecture anatomy model.
  3. Predict realistic failure modes across model, retrieval, workflow, UX, and operations.
  4. Design a narrow AI application of your own, including autonomy choice and prototype-versus-production thinking.

The intended learning outcome is not just “come up with an AI idea.” It is learning to think in terms of systems, layers, and tradeoffs from the very first module.
