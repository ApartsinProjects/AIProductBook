Big picture

AI engineering solves problems that are hard to fully encode as rules

Classical software

  • Procedure is explicit
  • Output is deterministic
  • Reliability comes from code + tests

AI-powered software

  • Goal and context are explicit
  • Behavior is probabilistic
  • Reliability comes from system design

Deep intuition

The model is not a source of truth. It is a flexible judgment component inside a larger engineered product.

Students should understand when AI belongs in the stack.3

Product types

Three common shapes appear first in most teams

Copilot

Human stays in the loop and uses AI for draft, lookup, or critique.

Knowledge system

Retrieve, ground, and answer from trusted sources.

Workflow

Classify, extract, route, summarize, or transform work at scale.

Connection

Shape is the first architecture decision, because it predicts context, controls, and evaluation style.

Product types in notes · Lab exercise 1
Categories matter because they imply different architectures.6
Product types

The other shapes push harder on autonomy and risk

Decision support

Improve human judgment where stakes are high and explanations matter.

Agent

Choose steps, call tools, and move work forward across a workflow.

Multimodal

Combine text with documents, images, audio, or live voice interaction.

Design implication

The more a system acts, the more approvals, observability, and recovery paths it needs.

These shapes raise the bar on trust and operational design.7

Architecture

Keep these three layers distinct

Product layer

User need, task, UX, autonomy, trust.

Application layer

Business logic, routing, context assembly, validation, logging.

Model layer

Generation, extraction, classification, tool use, reasoning.

Why it matters

Separate layers make design and debugging much clearer.

This slide prevents “the model did it” from becoming the whole explanation.10
Maturity

A demo proves possibility. Production proves trustworthiness.

Demo

Capability proof on ideal inputs.

Pilot

Workflow proof on real users and data.

Operated product

Measured, owned, observable, recoverable.

Production asks

  • How is quality measured?
  • How do we recover?
  • Who reviews risk?
  • What about cost and latency?

Deep insight

Production is a trust contract.

Prototype vs production in notes
This slide reframes maturity in operational terms.11
Failure modes

Start with the product-facing causes

Prompt

Instructions were weak or underspecified.

Context

The model saw the wrong inputs or too little context.

Retrieval

The right knowledge was not found or was ranked poorly.

Diagnostic discipline

Debug from the outside in: user symptom first, then system layer, then model.

Failure modes in notes · Lab exercise 3
Most visible failures are not “just the model.”12
Failure modes

Then check the system and operating conditions

Workflow

The surrounding logic made the wrong choice or skipped a check.

Model

The capability ceiling was too low for the task.

Operations

Latency, outage, or cost pressure broke the experience.

Teaching move

Every later module exists because one of these failure buckets becomes real in production.

This sets up later modules on evals, observability, and serving.13
Course logic

Every later module is really an answer to a failure mode

Prompting + structure

Reduce ambiguity.

RAG + retrieval

Reduce missing knowledge.

Workflows + agents

Turn output into action.

Evals + observability

Make quality visible.

Safety + approvals

Bound risk.

Serving + infra

Keep systems fast and available.

Concrete

What matters
