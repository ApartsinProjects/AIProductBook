Big Picture: Experimentation for Probabilistic Products

AI products require structured experimentation methods because the probabilistic nature of outputs makes it difficult to predict how changes will perform in production. A/B testing, canary deployments, interleaved testing, and shadow testing provide frameworks for measuring impact while managing variability. The key insight is using layered experimentation methods together for confidence at each stage.

Fun Note: The Canary Miners

Canary testing got its name from coal miners who brought canaries into mines. If the bird died, toxic gases were present. In software, we still use the canary metaphor even though nobody uses actual birds anymore.

7.3 Experimentation Methods

In Section 7.2, we covered eval-first PRDs and LLM-as-judge evaluation techniques. A key insight from that section was that AI outputs are probabilistic and can vary even with the same inputs. This variability makes it challenging to predict how changes will perform in production, which is why we need structured experimentation methods. Controlled experimentation is essential for AI products because the probabilistic nature of outputs makes it difficult to predict how changes will perform in production. Unlike traditional software where a code change produces deterministic results, an AI system may produce different outputs for the same input at different times. Experimentation methods provide frameworks for measuring the impact of changes while managing this variability.

A/B Testing for AI Products

A/B testing for AI products involves splitting traffic between two versions of the system and comparing outcomes. The key challenge is determining what to measure: are you optimizing for task completion rate, user satisfaction, or some other metric? A/B testing works well when you have sufficient traffic to achieve statistical significance and when the difference between versions is large enough to detect. For AI products with high variance in outputs, you may need much larger sample sizes than for traditional software experiments.

Case Study: The A/B Test That Changed Everything

A company tested two versions of their AI assistant: one that provided concise answers and one that provided detailed answers. They expected concise would win because users would save time. Instead, detailed answers won with 20% higher task completion. Users said they trusted the detailed answers more because they could see the reasoning. This insight changed their product strategy entirely: they optimized for thoroughness over brevity.

Skill: Designing AI A/B Tests

What to test in AI products:

  • Prompt variations: Different instructions to the AI
  • Model versions: New vs. old models
  • Temperature settings: Creativity vs. consistency
  • Output format: Short vs. long, structured vs. free-form
  • UI variations: How AI is presented to users

Key considerations:

  • Sample size: AI outputs have higher variance, requiring larger samples
  • Metrics: Choose metrics that reflect user value, not just AI quality
  • Duration: Run tests long enough to capture usage patterns
  • Segmentation: Analyze results by user type, not just aggregate

Canary Deployment

Canary deployment rolls out a new version to a small percentage of users before full deployment. This approach allows you to detect issues with real traffic before they affect all users. For AI products, canary deployments are particularly valuable because subtle quality issues may not appear in testing but become apparent with production traffic. The canary percentage is typically 5-10% initially, increasing if no issues are detected. Monitoring during the canary phase should include both technical metrics (latency, error rate) and quality metrics (task completion, user satisfaction).

Case Study: The Silent Quality Regression

A company deployed a model update to their AI chatbot. The technical metrics looked fine: latency, error rates, and throughput were all within normal ranges. However, canary deployment to 5% of users revealed a problem: user satisfaction scores dropped from 4.2 to 3.8. Further investigation showed the new model was producing slightly less helpful responses. If they had deployed to everyone at once, they would have degraded the experience for all users without immediately realizing why.

Skill: Canary Deployment for AI

Canary process:

  1. Deploy to 5% of users
  2. Monitor technical metrics: latency, errors, throughput
  3. Monitor quality metrics: task completion, user feedback
  4. If metrics are healthy, increase to 25%
  5. Continue monitoring, increase to 50%, then 100%
  6. If issues appear, rollback to previous version

Key metrics to monitor:

  • p50/p95 response time
  • Error rate by type
  • Task completion rate
  • User satisfaction (explicit and implicit feedback)
  • Helpful/unhelpful ratio

Interleaved Testing

Interleaved testing presents results from both versions mixed together, allowing direct comparison without requiring users to commit to one version. This method is particularly useful for search and recommendation systems where you can show results from both versions in a single interface. Users implicitly vote by which results they click on, providing immediate feedback on relative quality. Interleaved testing requires careful implementation to avoid confusing users, but it provides faster feedback than traditional A/B testing.

Case Study: Fast Feedback for Ranking

A search company wanted to test a new ranking algorithm. Traditional A/B testing would take weeks to get statistical significance. Instead, they implemented interleaved results: when users searched, they saw a mix of results from both the old and new algorithms. Users clicked on results from both algorithms, and the company could measure which algorithm won based on click-through rates. They got statistically significant results in days instead of weeks.

Shadow Testing

Shadow testing runs the new model in parallel with the production system without serving the new outputs to users. This approach lets you collect extensive comparison data with zero risk to users. Shadow testing is ideal for evaluating major model upgrades where you want to understand performance differences before exposing users to any risk. The limitation is that shadow traffic may not perfectly replicate production traffic patterns, so results are indicative rather than definitive.

Case Study: The Major Model Upgrade

A company wanted to upgrade from GPT-4 to a newer model. Rather than risking user experience, they ran shadow mode for two weeks: the new model processed all requests but its outputs were not shown to users. They compared outputs on hundreds of dimensions: accuracy, helpfulness, latency, cost. The analysis revealed the new model was faster and cheaper but slightly less accurate on technical questions. They used this insight to implement a hybrid approach: use the new model for most queries, but fall back to the old model for technical questions.

Experimentation Best Practice

Use multiple experimentation methods together. Run shadow tests before canary deployments. Use interleaved testing for rapid iteration on ranking. Use A/B testing for final validation. This layered approach gives you confidence at each stage while minimizing risk to users.
