16.5 Distributed Tracing and Resilience

Objective: Learn distributed tracing across AI pipelines and resilience patterns specific to AI workloads.

Prerequisites: Completion of Section 16.4 recommended. Familiarity with microservices architecture.
Big Picture: Design for Failure

This section covers distributed tracing across AI pipelines and resilience patterns specific to AI workloads. In the context of AI engineering, failure is not an exception but the default state in distributed systems. Designing for failure ensures your AI product continues operating (at reduced quality) when components fail.

Fun Note: The Circuit Breaker That Breaks Too Often

A team implemented a circuit breaker that opened after 5 failures. Their AI had natural variance, so 5 failures happened weekly during normal operation. The circuit breaker was tripping constantly, degrading service unnecessarily. The fix: increase the threshold to match actual failure patterns during normal operation.

16.5 Distributed Tracing and Resilience

Production AI systems span multiple services: API gateway, retrieval pipeline, model serving, post-processing. When something goes wrong, you need to trace a request across all these services to find the failure point. Distributed tracing gives you that visibility. Resilience patterns ensure that failures in one component do not cascade.

Why This Matters

In a microservices architecture, a single user request might touch 10+ services. If latency spikes or errors increase, you need to know which service is the culprit. Without distributed tracing, this is guesswork. With tracing, it is obvious. Similarly, without resilience patterns, a single service failure brings down your entire AI capability.

1. Distributed Tracing for AI Pipelines

OpenTelemetry has become the standard for distributed tracing. It provides instrumentation that works across services and language runtimes, including Python for AI workloads.

Instrumenting a RAG Pipeline
from opentelemetry import trace
from opentelemetry.trace import Status, StatusCode

class TracedRAGPipeline:
    """RAG pipeline with full distributed tracing"""
    
    def __init__(self, retriever, ranker, llm):
        self.retriever = retriever
        self.ranker = ranker
        self.llm = llm
        self.tracer = trace.get_tracer(__name__)
    
    async def answer(self, query: str, trace_context=None):
        # Extract any incoming trace context for propagation
        ctx = trace_context and trace_context.extract()
        
        with self.tracer.start_as_current_span(
            "rag_pipeline",
            context=ctx,
            kind=trace.SpanKind.INTERNAL
        ) as span:
            # Add query attributes
            span.set_attribute("query.length", len(query))
            span.set_attribute("query.hash", hash(query))
            
            # Step 1: Retrieval
            with self.tracer.start_span("retrieval", parent=span) as retrieval_span:
                retrieval_start = time.time()
                docs = await self.retriever.invoke(query)
                retrieval_span.set_attribute("num_docs_retrieved", len(docs))
                retrieval_span.set_attribute("retrieval.duration_ms", time.time() - retrieval_start)
                retrieval_span.set_status(Status(StatusCode.OK if docs else StatusCode.ERROR))
            
            # Step 2: Reranking
            with self.tracer.start_span("reranking", parent=span) as rerank_span:
                rerank_start = time.time()
                reranked = await self.ranker.rerank(query, docs, top_k=5)
                rerank_span.set_attribute("num_docs_reranked", len(reranked))
                rerank_span.set_attribute("rerank.duration_ms", time.time() - rerank_start)
            
            # Step 3: Generation
            with self.tracer.start_span("generation", parent=span) as gen_span:
                gen_start = time.time()
                context = self.format_context(reranked)
                answer = await self.llm.generate(query, context)
                gen_span.set_attribute("answer.length", len(answer))
                gen_span.set_attribute("generation.duration_ms", time.time() - gen_start)
                
                # Add token usage if available
                if hasattr(answer, 'usage'):
                    gen_span.set_attribute("tokens.prompt", answer.usage.prompt_tokens)
                    gen_span.set_attribute("tokens.completion", answer.usage.completion_tokens)
            
            span.set_status(Status(StatusCode.OK))
            return answer

2. Trace Context Propagation

AI systems often span multiple services. Trace context propagation ensures a single trace ID follows requests across service boundaries.

Context Propagation Across Services
from opentelemetry.propagate import inject, extract
from opentelemetry.trace import Span, SpanKind

async def call_model_service(user_query: str, trace_context: dict):
    headers = {}
    
    # Inject current trace context into outgoing headers
    inject(headers)
    
    # downstream call carries trace context
    response = await http_client.post(
        "http://model-service/generate",
        json={"query": user_query},
        headers={
            **headers,
            "trace-context": str(headers)  # Implementation varies
        }
    )
    
    return response

async def model_service_handler(request):
    # Extract incoming trace context
    ctx = extract(request.headers)
    
    with tracer.start_as_current_span(
        "model_service_generate",
        context=ctx,
        kind=SpanKind.SERVER  # This is a server receiving a request
    ) as span:
        span.set_attribute("internal.request_id", request.id)
        
        # Process with full trace context
        result = await generate_response(request.json(), span)
        
        return result

3. AI-Specific Resilience Patterns

AI workloads have unique resilience challenges. LLM calls are slow (hundreds of milliseconds to seconds), expensive, and can return variable quality. Standard resilience patterns must be adapted for these characteristics.

Circuit Breaker for LLM Calls
from circuitbreaker import circuit
import time

class CircuitBreakerLLMClient:
    """LLM client with circuit breaker pattern"""
    
    def __init__(self, base_client, failure_threshold=5, recovery_timeout=60):
        self.client = base_client
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.failures = 0
        self.last_failure_time = None
        self.state = "CLOSED"  # CLOSED, OPEN, HALF_OPEN
    
    async def generate(self, prompt: str, **kwargs):
        # Check circuit state
        if self.state == "OPEN":
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = "HALF_OPEN"
            else:
                raise CircuitBreakerOpenError("Circuit breaker is OPEN")
        
        try:
            result = await self.client.generate(prompt, **kwargs)
            self._on_success()
            return result
        
        except Exception as e:
            self._on_failure()
            raise
    
    def _on_success(self):
        self.failures = 0
        self.state = "CLOSED"
    
    def _on_failure(self):
        self.failures += 1
        self.last_failure_time = time.time()
        
        if self.failures >= self.failure_threshold:
            self.state = "OPEN"
            print(f"Circuit breaker opened after {self.failures} failures")
Bulkhead Isolation for AI Workloads
from asyncio import Semaphore
from concurrent.futures import ThreadPoolExecutor

class BulkheadPattern:
    """
    Bulkhead pattern: isolate different workload types into separate resource pools.
    Prevents one workload type from starving others.
    """
    
    def __init__(self):
        # Separate semaphores for different workload types
        self.simple_tasks = Semaphore(50)   # Simple Q&A: high concurrency OK
        self.agent_tasks = Semaphore(10)   # Agent tasks: expensive, limit concurrency
        self.batch_tasks = Semaphore(5)     # Batch processing: GPU intensive
    
    async def execute_simple(self, task_fn, *args, **kwargs):
        async with self.simple_tasks:
            return await task_fn(*args, **kwargs)
    
    async def execute_agent(self, task_fn, *args, **kwargs):
        async with self.agent_tasks:
            return await task_fn(*args, **kwargs)
    
    async def execute_batch(self, task_fn, *args, **kwargs):
        async with self.batch_tasks:
            return await task_fn(*args, **kwargs)

# Usage: Route tasks to appropriate bulkhead
bulkhead = BulkheadPattern()

async def handle_request(request):
    if request.type == "simple_qa":
        return await bulkhead.execute_simple(process_simple, request)
    elif request.type == "agent_research":
        return await bulkhead.execute_agent(process_agent, request)
    elif request.type == "batch_summary":
        return await bulkhead.execute_batch(process_batch, request)

4. Timeout Patterns for AI Workloads

LLM calls need different timeout strategies than standard HTTP requests. Short timeouts cause unnecessary failures; long timeouts cause slow error reporting.

Adaptive Timeout Strategy
class AdaptiveTimeoutStrategy:
    """
    Timeouts that adapt based on expected latency for different operations.
    """
    
    # Base timeouts by operation type (in seconds)
    BASE_TIMEOUTS = {
        "embedding": 5,
        "retrieval": 2,
        "reranking": 3,
        "simple_generation": 15,
        "complex_generation": 60,
        "agent_iteration": 30,
    }
    
    def __init__(self, latency_percentiles: dict = None):
        self.percentiles = latency_percentiles or {}
    
    def get_timeout(self, operation: str, num_docs: int = 0, estimated_complexity: str = "medium") -> float:
        base = self.BASE_TIMEOUTS.get(operation, 30)
        
        # Adjust for retrieval size
        if operation == "retrieval" and num_docs > 0:
            base = base * (1 + num_docs / 100)
        
        # Adjust for complexity
        if operation == "generation":
            if estimated_complexity == "high":
                base *= 2
            elif estimated_complexity == "low":
                base *= 0.5
        
        # Adjust for observed latency patterns
        if operation in self.percentiles:
            observed_p99 = self.percentiles[operation]
            # Use observed p99 + 50% headroom
            base = max(base, observed_p99 * 1.5)
        
        return base

# Usage
timeouts = AdaptiveTimeoutStrategy({"generation": 12.0})

async def generate_with_timeout(prompt: str, context: str):
    timeout = timeouts.get_timeout(
        "generation",
        estimated_complexity="high"
    )
    
    try:
        return await asyncio.wait_for(
            llm.generate(prompt, context),
            timeout=timeout
        )
    except asyncio.TimeoutError:

5. Graceful Degradation Strategies

AI systems should degrade gracefully, not fail completely. When the ideal path is unavailable, provide the best available response.

Multi-Tier Degradation
class GracefulDegradation:
    """
    Multi-tier degradation: Provide best available response as services degrade.
    """
    
    async def answer(self, query: str) -> str:
        # Tier 1: Full RAG pipeline (best quality)
        try:
            return await self.tier1_full_rag(query)
        
        except VectorDBError:
            # Tier 2: Keyword search fallback
            try:
                return await self.tier2_keyword_fallback(query)
            except Exception:
                pass
        
        except TimeoutError:
            # Tier 3: Cached similar response
            try:
                return await self.tier3_cache_fallback(query)
            except Exception:
                pass
        
        except Exception as e:
            # Tier 4: Simple generation without retrieval
            try:
                return await self.tier4_simple_generation(query)
            except Exception:
                pass
        
        # Final fallback: Polite error
        return self.tier5_error_response()
    
    async def tier1_full_rag(self, query):
        docs = await self.vector_db.similarity_search(query)
        reranked = await self.ranker.rerank(query, docs)
        return await self.llm.generate(query, reranked)
    
    async def tier2_keyword_fallback(self, query):
        docs = await self.keyword_index.search(query)
        return await self.llm.generate(query, docs)
    
    async def tier3_cache_fallback(self, query):
        cached = await self.cache.get_similar(query)
        if cached:
            return cached.response
        raise CacheMissError()
    
    async def tier4_simple_generation(self, query):
        # No retrieval, rely on model's knowledge
        return await self.llm.generate(f"Answer from general knowledge: {query}")
    
    def tier5_error_response(self):
        return "I apologize, but I'm experiencing technical difficulties. Please try again."

6. Health Checks for AI Services

Standard health checks verify a service is running. AI services need deeper checks: model loaded, GPU available, dependencies accessible.

AI Service Health Check
class AIHealthCheck:
    """
    Multi-level health check for AI services.
    """
    
    def __init__(self, model_loader, vector_db, gpu_available):
        self.model_loader = model_loader
        self.vector_db = vector_db
        self.gpu_available = gpu_available
    
    async def check(self) -> dict:
        checks = {}
        
        # Level 1: Basic service health
        checks["service_alive"] = True
        
        # Level 2: GPU availability
        checks["gpu_available"] = self.gpu_available.check()
        
        # Level 3: Model loaded and responsive
        checks["model_loaded"] = await self._check_model()
        
        # Level 4: Dependencies accessible
        checks["vector_db_healthy"] = await self._check_vector_db()
        
        # Level 5: Latency under threshold
        checks["latency_acceptable"] = await self._check_latency()
        
        # Determine overall health
        critical_checks = [
            checks["service_alive"],
            checks["gpu_available"],
            checks["model_loaded"]
        ]
        
        checks["healthy"] = all(critical_checks)
        checks["degraded"] = any(checks.values()) and not checks["healthy"]
        
        return checks
    
    async def _check_model(self) -> bool:
        try:
            start = time.time()
            # Lightweight generation test
            response = await self.model_loader.test_generate("Hi")
            latency = time.time() - start
            return latency < 5.0  # Should respond within 5 seconds
        except:
            return False
    
    async def _check_vector_db(self) -> bool:
        try:
            await self.vector_db.ping()
            return True
        except:
            return False
    
    async def _check_latency(self) -> bool:
        try:
            start = time.time()
            await self.model_loader.test_generate("Test")
            latency = time.time() - start
            return latency < 10.0  # p99 should be under 10s
        except:
            return True  # Don't fail health check for latency alone

Design for Failure

In distributed AI systems, failure is not an exception, it is the default state. Some component will be unavailable at any given time. Design your system to continue operating (possibly at reduced quality) when parts fail. The difference between a resilient AI product and a brittle one is whether you planned for failure before it happened.
