16.1 Latency, Throughput, Multi-Tenancy

Production AI systems must handle real-world demands: low latency for user satisfaction, high throughput for cost efficiency, and isolation for multi-tenant security. Understanding these requirements and how to meet them is essential for production deployment.

Why This Matters

Users are sensitive to latency. Every 100ms of delay increases abandonment rates. At scale, throughput determines whether your system is cost-effective. Multi-tenancy ensures one customer's issues do not affect others. These are table-stakes for production systems.

Academic Perspective: Observability and the Three Pillars

The concept of observability in AI systems draws from control theory and modern SRE practice. Originating from dynamical systems theory (Kalman, 1960), observability measures how well you can infer internal states from external outputs. For AI systems, this translates to understanding model behavior, retrieval quality, and generation characteristics from logs and metrics.

Tan et al.'s Three Pillars of Observability framework (metrics, logs, traces) extends naturally to AI systems with a fourth pillar: evaluation signals. Unlike traditional software where behavior is deterministic and observable through standard telemetry, AI systems require purpose-built evaluation infrastructure to understand whether outputs are correct, appropriate, and useful. Production AI observability must capture both operational metrics (latency, throughput) and quality metrics (relevance, hallucination rates, user satisfaction signals).

Key references: Kalman, R.E. (1960). "On the General Theory of Control." IFAC; Tan, B. et al. (2022). "The Three Pillars of Observability." SRE Weekly.

Rule of Thumb: The 2-Second Target

Target p99 under 2 seconds for chat, 500ms for autocomplete.

Your users live in the tail, not the average. Optimize for p99, not average latency. The most frustrated users are the ones who wait longest.

Watch Out: p99 Latency vs Average Latency

A common mistake is optimizing for average latency while ignoring tail latency. In AI systems, the average might be good (1.5s) but p99 could be terrible (10s) because of queuing, retries, or slow model inference. Always measure and target p99 or p95 latency, not averages. Your most frustrated users are in the tail, not the average.

1. Latency Management

Target p99 latency under 2 seconds for chat interfaces. This means 99% of requests must complete within 2 seconds, providing a consistently responsive experience.

Practical Example: Latency Optimization
class LatencyOptimizer:
    """Optimize for consistent low latency"""
    
    def __init__(self):
        self.cache = RedisCache()
    
    async def process_request(self, query: str) -> Response:
        start = time.time()
        
        # Check cache first
        cached = await self.cache.get(query)
        if cached:
            return Response(text=cached, from_cache=True, latency=time.time() - start)
        
        # Retrieval should be fast (<200ms for most cases)
        docs = await self.retriever.aget(query, k=3)  # Limit docs for speed
        
        # Generate response
        response = await self.llm.agenerate(
            docs=docs,
            query=query,
            max_tokens=500  # Limit output length
        )
        
        # Cache the result
        await self.cache.set(query, response.text)
        
        total_latency = time.time() - start
        return Response(text=response.text, latency=total_latency)

# Latency breakdown targets:
# - Cache lookup: <10ms
# - Retrieval: <200ms
# - Generation: <1500ms (depends on model)
# - Total: <2000ms p99

Latency Optimization Techniques

  • Streaming: Start returning tokens before full response is ready
  • Prefetching: Anticipate next user action and prepare in advance
  • Caching: Cache common queries and responses
  • Batching: Batch requests for better GPU utilization

2. Throughput Planning

Plan for 10x traffic growth. Systems that work at 100 requests/day often fail at 1,000. Design capacity with headroom.

3. Multi-Tenancy Isolation

In multi-tenant systems, you must isolate tenant data and compute. One tenant's workload shouldn't affect another's performance or security.

Skill Demonstration: Tenant Isolation
class TenantAwareSystem:
    """Ensure tenant isolation at every layer"""
    
    def __init__(self):
        self.rate_limiters = {}  # Per-tenant rate limits
        self.resource_pools = {}  # Per-tenant compute allocation
    
    async def process(self, tenant_id: str, query: str) -> Response:
        # Check rate limits
        if not self.check_rate_limit(tenant_id):
            raise RateLimitExceeded(tenant_id)
        
        # Get tenant-specific resources
        resource = self.get_tenant_resources(tenant_id)
        
        # Use tenant-specific data (no cross-tenant data access)
        docs = await self.get_tenant_docs(tenant_id, query)
        
        # Generate with tenant context
        response = await self.llm.agenerate(
            docs=docs,
            query=query,
            system_prompt=self.get_tenant_prompt(tenant_id)
        )
        
        # Log for tenant-specific monitoring
        self.log_request(tenant_id, response)
        
        return response
    
    def check_rate_limit(self, tenant_id: str) -> bool:
        # Each tenant gets their own rate limit bucket
        limiter = self.rate_limiters.get(tenant_id)
        if not limiter:
            limiter = TokenBucket(rate=1000, capacity=1000)
            self.rate_limiters[tenant_id] = limiter
        return limiter.consume(1)
    
    async def get_tenant_docs(self, tenant_id: str, query: str):
        # Ensure only tenant's documents are retrieved
        return await self.vectorstore.as_retriever(
            filter={"tenant_id": tenant_id}  # Database-level isolation
        ).aget(query)
