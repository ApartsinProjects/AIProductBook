13.2 Chunking, Embeddings, Hybrid Search

The quality of your RAG system depends heavily on how you prepare your data before retrieval. Three key decisions shape retrieval quality: how you chunk documents into searchable units, which embedding model you use, and how you combine different retrieval methods.

Why This Matters

These decisions have a larger impact on retrieval quality than any other factor. You can spend weeks optimizing your retrieval pipeline, but if your chunking strategy is wrong, nothing will work. The key insight is that there is no universal optimal solution; the right approach depends on your data characteristics and query patterns.

Watch Out: There's No Perfect Chunk Size

You'll often see recommendations like "use 512 tokens" as if it's universal. This is misleading. The optimal chunk size depends on your data structure, query types, and embedding model. A good starting point exists, but you must test with your actual data. Chunk sizes that work well for one use case can completely fail for another.

1. Chunking Strategies

Chunking determines the granularity of your search. Too small, and you lose context; too large, and precision suffers. The optimal chunk size typically falls between 256 and 1024 tokens, though this varies by use case.

Case Study: Chunking for Different Use Cases

Q&A Application:

For question-answering, smaller chunks (256-512 tokens) work better because they align with the size of answer spans. Users ask specific questions, and the relevant information fits in small chunks.

from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,  # Small overlap for Q&A
    length_function=len,
)
chunks = splitter.split_documents(docs)

Document Summarization:

For summarization, larger chunks (1024-2048 tokens) preserve more context and allow the model to see the full picture.

splitter = RecursiveCharacterTextSplitter(
    chunk_size=2000,
    chunk_overlap=200,  # Larger overlap for context
)

Advanced Chunking Strategies

Semantic Chunking: Split at natural breakpoints rather than fixed sizes. Uses embeddings to find boundaries where meaning changes.

from langchain_experimental.text_splitter import SemanticChunker
from langchain_openai import OpenAIEmbeddings

semantic_splitter = SemanticChatGLMTextSplitter(
    embeddings=OpenAIEmbeddings(),
    breakpoint_threshold=0.7,  # How different must adjacent chunks be
)
chunks = semantic_splitter.split_documents(docs)

Hierarchical Chunking: Create multiple levels of granularity for different query types.

Skill Demonstration: Hierarchical Chunking
class HierarchicalChunker:
    """Create multiple chunk sizes for flexible retrieval"""
    
    def __init__(self):
        self.large_splitter = RecursiveCharacterTextSplitter(
            chunk_size=2000, overlap=200
        )
        self.medium_splitter = RecursiveCharacterTextSplitter(
            chunk_size=500, overlap=50
        )
    
    def chunk(self, documents):
        results = []
        for doc in documents:
            # Create large chunks for context
            large_chunks = self.large_splitter.split_documents([doc])
            # Create medium chunks for precision
            medium_chunks = self.medium_splitter.split_documents([doc])
            
            for i, chunk in enumerate(large_chunks):
                chunk.metadata["chunk_level"] = "large"
                chunk.metadata["chunk_index"] = i
                results.append(chunk)
            
            for i, chunk in enumerate(medium_chunks):
                chunk.metadata["chunk_level"] = "medium"
                chunk.metadata["chunk_index"] = i
                results.append(chunk)
        
        return results

# Now you can retrieve at the appropriate level:
# - Use large chunks when queries need broad context
# - Use medium chunks when queries are specific

2. Embedding Model Selection

Embedding models convert text into numerical vectors that capture semantic meaning. The choice of embedding model significantly impacts retrieval quality. Modern embedding models like OpenAI's text-embedding-3 series and Cohere's embedding models offer strong performance across use cases.

For multilingual applications, consider models specifically trained on cross-lingual data. The key metrics to evaluate are recall (does it retrieve all relevant documents) and precision (does it exclude irrelevant documents).

3. Hybrid Search

Hybrid search combines multiple retrieval methods for better results than any single method alone. The most common combination is BM25 (a traditional keyword-based method) with dense vector retrieval (semantic similarity).

Practical Example: Implementing Hybrid Search
from langchain.retrievers import EnsembleRetriever
from langchain_community.retrievers import BM25Retriever
from langchain_openai import OpenAIEmbeddings
from langchain_pinecone import PineconeVectorStore

# Dense retriever (semantic)
vectorstore = PineconeVectorStore.from_existing_index(...)
dense_retriever = vectorstore.as_retriever(search_kwargs={"k": 10})

# Sparse retriever (keyword-based)
bm25_retriever = BM25Retriever.from_documents(documents)
bm25_retriever.k = 10

# Combine with weighting
ensemble_retriever = EnsembleRetriever(
    retrievers=[dense_retriever, bm25_retriever],
    weights=[0.6, 0.4]  # 60% semantic, 40% keyword
)

# Query
results = ensemble_retriever.invoke("What is the return policy?")

Why Hybrid Works

  • BM25 excels at finding exact matches and handling rare terms that may not be well-represented in embeddings
  • Dense retrieval excels at finding semantically similar results even when keywords do not match
  • Combining them captures both signals, improving recall

4. Reranking

After retrieval, reranking with a cross-encoder model reorders the top results based on more accurate relevance assessment, trading some speed for precision on the most important results.

Code Example: Adding Reranking
from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import CrossEncoderReranker
from langchain_community.cross_encoders import CrossEncoder

# Base retriever (whatever you use)
base_retriever = vectorstore.as_retriever(k=20)

# Cross-encoder for reranking
cross_encoder = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

# Wrap with reranker
compression_retriever = ContextualCompressionRetriever(
    base_compressor=CrossEncoderReranker(
        cross_encoder=cross_encoder,
        top_n=5  # Return top 5 after reranking
    ),
    base_retriever=base_retriever
)

# Now results are reranked by actual relevance
results = compression_retriever.invoke("What is the return policy?")
