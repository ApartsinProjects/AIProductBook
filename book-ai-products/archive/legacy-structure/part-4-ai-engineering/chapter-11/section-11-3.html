11.3 Layers of the AI Stack

Understanding the layers of the AI stack helps you make informed decisions about where to invest effort and when to use existing solutions versus building custom components. Each layer has emerged to address specific challenges in building AI products, and the ecosystem around each layer is maturing rapidly.

Why This Matters

The AI stack is still evolving, but understanding these layers helps you architect systems that are both effective today and adaptable to future changes. The key insight is that you should leverage existing solutions at most layers while maintaining the flexibility to swap them out. The rapid pace of innovation means today's best solution might be tomorrow's outdated choice.

Watch Out: Don't Over-Engineer Your Stack

A common mistake is building custom solutions at every layer "just in case." The reality is that managed services at most layers are good enough for most products. Start with managed solutions (API providers, managed vector DBs, hosted inference) and only build custom components when you hit specific limitations. Premature optimization at multiple layers adds unnecessary complexity.

The Five Layers

1. Compute Layer

The compute layer encompasses the GPU infrastructure and cloud resources that power AI model execution. Today, most teams do not own GPU infrastructure but rent it from cloud providers. The key decisions at this layer involve selecting the right instance types for your workload, optimizing for cost-performance tradeoffs, and understanding the differences between model serving and training requirements.

Real-World Scenario: Choosing Compute Options

A startup building a document analysis product faced a choice:

  • AWS SageMaker: Fully managed, easy setup, predictable costs
  • Azure ML: Strong enterprise integration, good for Microsoft-heavy teams
  • Google Vertex: Best for training, strong TPUs available
  • Self-hosted on Lambda Labs: Cheaper at scale, more operational work
  • RunPod/Vast.ai: Flexible spot instances, cost-effective for batch work

They chose SageMaker for production serving (reliability priority) and Lambda Labs for fine-tuning (cost priority). Multi-cloud strategies are increasingly common as teams seek to avoid vendor lock-in and optimize costs.

2. Models Layer

The models layer includes foundation models from providers like OpenAI, Anthropic, Google, and open-source alternatives, along with fine-tuned custom models. The dominant strategy today is to start with foundation models and only invest in fine-tuning when specific use cases require it. Foundation models improve rapidly, making custom fine-tuning a moving target.

Case Study: Model Selection for a Legal Tech Product

A legal tech company needed to summarize contracts. They evaluated multiple options:

The key decision is which model to use for each task: generally, use the most capable model you can afford for complex reasoning, and smaller models for simple tasks where cost matters more.

3. Data and Retrieval Layer

The data and retrieval layer is critical for products that use RAG (retrieval-augmented generation) architectures. This includes vector databases like Pinecone, Weaviate, and pgvector, along with the RAG pipelines that manage document processing, embedding generation, and retrieval. RAG optimization is a deep topic that involves understanding chunking strategies, embedding model selection, and retrieval algorithm tuning. The quality of retrieval directly impacts the quality of generation, making this layer essential for RAG products.

Skill Demonstration: Building a Retrieval Pipeline
from langchain_community.retrievers import BM25Retriever
from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import CrossEncoderReranker
from langchain_community.cross_encoders import CrossEncoder

class AdvancedRetrievalPipeline:
    """Combines multiple retrieval strategies for better results"""
    
    def __init__(self, documents: list, vectorstore):
        # Dense retrieval (semantic similarity)
        self.dense_retriever = vectorstore.as_retriever(
            search_type="similarity_threshold",
            search_kwargs={"k": 10, "score_threshold": 0.7}
        )
        
        # Sparse retrieval (keyword-based BM25)
        self.bm25_retriever = BM25Retriever.from_documents(documents)
        self.bm25_retriever.k = 10
        
        # Reranking model
        self.cross_encoder = CrossEncoder(
            "cross-encoder/ms-marco-MiniLM-L-6-v2"
        )
    
    def retrieve(self, query: str) -> list:
        # Get results from both retrievers
        dense_results = self.dense_retriever.invoke(query)
        bm25_results = self.bm25_retriever.invoke(query)
        
        # Combine and deduplicate
        combined = self._merge_results(dense_results, bm25_results)
        
        # Rerank top results
        reranked = self._rerank(query, combined[:20])
        
        return reranked[:5]  # Return top 5 after reranking
    
    def _merge_results(self, dense: list, bm25: list) -> list:
        # Deduplicate and interleave results
        seen = set()
        merged = []
        for d, b in zip(dense, bm25):
            if d.page_content not in seen:
                seen.add(d.page_content)
                merged.append(d)
            if b.page_content not in seen:
                seen.add(b.page_content)
                merged.append(b)
        return merged
    
    def _rerank(self, query: str, docs: list) -> list:
        # Score all documents with cross-encoder
        pairs = [(query, doc.page_content) for doc in docs]
        scores = self.cross_encoder.predict(pairs)
        
        # Sort by score and return
        scored_docs = list(zip(docs, scores))
        scored_docs.sort(key=lambda x: x[1], reverse=True)
        return [doc for doc, score in scored_docs]

4. Orchestration Layer

The orchestration layer manages the flow of information through your AI system. For simple applications, this may just be API calls. For complex applications, orchestration frameworks like LangGraph, CrewAI, and AutoGen provide tools for building multi-agent systems, managing state, and handling complex workflows. The choice of orchestration framework affects how quickly you can build new features and how maintainable your codebase becomes.

Real-World Scenario: Choosing an Orchestration Framework

A team building a research assistant evaluated orchestration options:

FrameworkBest ForTradeoffs
LangGraphComplex workflows with cycles, stateful appsSteeper learning curve, very flexible
CrewAIRole-based multi-agent systemsMore opinionated, good for team simulations
AutoGenFlexible agent communication patternsMicrosoft ecosystem, extensive but complex
Direct API callsSimple, linear workflowsLimited to single-step operations

They chose LangGraph because they needed to handle complex research workflows that required iteration and conditional branching.

5. Tooling Layer

The tooling layer encompasses the infrastructure for observability, evaluation, and continuous integration. This includes tracing tools like LangSmith and Langfuse, evaluation frameworks like LangChain evals, and CI/CD systems adapted for AI models. The tooling layer is what makes your system production-ready, enabling you to monitor quality, detect regressions, and deploy updates safely.
