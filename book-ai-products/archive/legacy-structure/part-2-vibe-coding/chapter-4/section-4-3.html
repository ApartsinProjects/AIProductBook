4.3 Transitioning to Testable Code

Objective: Teach practical, repeatable workflows for moving from vibe to production.

Prerequisites: Completion of Section 4.2 recommended. Basic understanding of testing concepts.
Big Picture: From Prototype to Production

This section covers the transformation from AI-generated code to production-ready, testable code. In the context of vibe coding, this transition is essential because initial AI outputs often lack the type safety, error handling, and testability that production requires. These patterns transform prototyping into genuine development.

4.3 Transitioning to Testable Code

Vibe-generated code often needs refinement to be production-ready. While AI excels at generating code that follows patterns, it sometimes produces code that is difficult to test, maintain, or extend. These issues are not apparent in initial generation but become problematic over time as the codebase grows. This section covers the specific transformations that make AI-generated code ready for production deployment.

The gap between "works on my machine" and "production-ready" is often invisible during initial development. Your AI assistant generates code that executes correctly for the happy path, but production environments expose weaknesses that development testing misses. Network failures, unexpected data formats, edge cases in user input, and concurrent operations all stress code in ways that initial generation does not anticipate. Transitioning to testable code is not about adding tests after the fact; it is about writing code that can be tested reliably and refactored confidently.

4.3.1 Type Safety: The Foundation of Production Code

Adding TypeScript types is often the first priority because type safety catches bugs at compile time rather than runtime. AI-generated code may have implicit types that work but are not explicitly declared, leading to potential errors when the code is modified. Go through the generated code and ensure full type safety with explicit interfaces for all data structures, proper generics where appropriate, and strict null checks enabled. This investment pays dividends in maintainability and refactoring safety.

Why This Matters: Type errors in production cause runtime failures that affect real users. A missing null check that works in development may crash your application when a database query returns empty. TypeScript catches these issues before deployment, saving hours of debugging and preventing customer-facing incidents.

Consider a typical AI-generated function that processes user data. The AI might generate something that works but lacks explicit typing:

Before: Implicit Types (Difficult to Test)
// AI-generated code - works but lacks type safety
function processUserData(users) {
    return users.map(user => ({
        id: user.id,
        displayName: user.name || user.email,
        initials: user.name.split(' ').map(n => n[0]).join(''),
        createdAt: new Date(user.created)
    }));
}

This code has multiple failure points. If user.name is null, the code crashes. If user.created is missing, the Date constructor fails. The function signature does not tell callers what input it expects or what it returns. Here is the transformed version:

After: Explicit Types (Testable and Maintainable)
interface UserRecord {
    id: string;
    name?: string | null;
    email: string;
    created?: string | null;
}

interface ProcessedUser {
    id: string;
    displayName: string;
    initials: string;
    createdAt: Date;
}

/**
 * Processes raw user records into display-ready user objects.
 * @param users - Array of raw user records from the database
 * @returns Array of processed users with computed fields
 * @throws Error if user record has invalid email
 */
function processUserData(users: UserRecord[]): ProcessedUser[] {
    return users.map(user => {
        if (!user.email) {
            throw new Error(`User ${user.id} has no email address`);
        }

        const displayName = user.name?.trim() || user.email;
        const initials = displayName
            .split(' ')
            .filter(part => part.length > 0)
            .map(part => part[0].toUpperCase())
            .join('');

        return {
            id: user.id,
            displayName,
            initials,
            createdAt: user.created ? new Date(user.created) : new Date()
        };
    });
}

The typed version is longer, but it provides several advantages. The interface documents the expected shape of input and output. The function signature tells TypeScript what it accepts and returns, enabling autocomplete and compile-time error detection. The error handling makes failure cases explicit rather than letting them propagate as cryptic runtime errors. This code can now be unit tested with confidence.

Skill Demonstration: Adding Type Safety to AI Code

Follow these steps when refining AI-generated code:

  1. Identify data boundaries. Find where your code interacts with external systems: API inputs, database results, configuration files. These are where type mismatches cause production bugs.
  2. Define interfaces first. Before refactoring functions, write interfaces that describe your data. This forces you to think about what data you actually have versus what you assume you have.
  3. Enable strict mode. In your TypeScript config, set strict: true and noImplicitAny: true. This prevents the AI from generating code that falls back to implicit any types.
  4. Add generic types. For functions that work with multiple data types, use generics to maintain type safety while preserving flexibility.
  5. Test at boundaries. Write tests that verify your type assertions match reality. If a test fails because the actual data shape differs from your interface, your interface was wrong.

4.3.2 Extracting Logic: From Components to Pure Functions

Extracting logic from large components into testable functions improves both testability and reusability. AI often generates code as a single block that is difficult to unit test because the logic is tangled with presentation. Extract business logic into pure functions that take inputs and return outputs without side effects, making them trivial to test. Similarly, add error handling to make edge cases explicit rather than implicit. Document interfaces with JSDoc comments so that future developers (and AI) can understand the contract each function fulfills.

Why This Matters: tangled logic inside UI components cannot be unit tested in isolation. You must render the entire component to test business logic, which is slow and brittle. Pure functions can be tested in milliseconds without any framework. This separation also enables code reuse across components and even across projects.

AI frequently generates React components or similar UI frameworks with business logic embedded directly in the component. While this works for initial development, it creates testing challenges. Consider this AI-generated component:

Before: Business Logic Entangled in Component
function UserDashboard({ userIds }) {
    const [users, setUsers] = useState([]);
    const [loading, setLoading] = useState(true);
    const [error, setError] = useState(null);

    useEffect(() => {
        async function loadUsers() {
            setLoading(true);
            setError(null);
            try {
                const results = await Promise.all(
                    userIds.map(id => fetch(`/api/users/${id}`).then(r => r.json()))
                );
                
                const processed = results.map(user => {
                    // Business logic buried in component
                    const isPremium = user.subscription === 'pro' || 
                                      user.subscription === 'enterprise';
                    const renewalDate = new Date(user.renewal);
                    const daysUntilRenewal = Math.ceil(
                        (renewalDate - new Date()) / (1000 * 60 * 60 * 24)
                    );
                    
                    return {
                        ...user,
                        isPremium,
                        daysUntilRenewal,
                        needsAttention: daysUntilRenewal <= 30 && !user.hasPaymentMethod
                    };
                });
                
                setUsers(processed);
            } catch (err) {
                setError(err.message);
            } finally {
                setLoading(false);
            }
        }
        
        loadUsers();
    }, [userIds]);

    if (loading) return <Spinner />;
    if (error) return <ErrorMessage message={error} />;
    
    return (
        <div>
            {users.map(user => (
                <UserCard key={user.id} user={user} />
            ))}
        </div>
    );
}

The business logic for calculating premium status, renewal dates, and attention flags is embedded in the useEffect callback. To test this logic, you must render the entire component, mock the fetch API, and wait for effects to complete. Here is the refactored version with extracted pure functions:

After: Extracted Pure Functions
// Pure function - completely testable without any framework
interface RawUser {
    id: string;
    name: string;
    subscription: string;
    renewal: string;
    hasPaymentMethod: boolean;
}

interface UserWithFlags {
    id: string;
    name: string;
    isPremium: boolean;
    daysUntilRenewal: number;
    needsAttention: boolean;
}

/**
 * Determines if a user has a premium subscription.
 * @param subscription - The subscription tier
 * @returns true if user has pro or enterprise subscription
 */
function isPremiumUser(subscription: string): boolean {
    return subscription === 'pro' || subscription === 'enterprise';
}

/**
 * Calculates days until subscription renewal.
 * @param renewalDate - ISO date string of renewal date
 * @returns Number of days until renewal (negative if past due)
 */
function calculateDaysUntilRenewal(renewalDate: string): number {
    const renewal = new Date(renewalDate);
    const now = new Date();
    const msPerDay = 1000 * 60 * 60 * 24;
    return Math.ceil((renewal.getTime() - now.getTime()) / msPerDay);
}

/**
 * Determines if a user's subscription needs attention.
 * @param daysUntilRenewal - Days until renewal (can be negative)
 * @param hasPaymentMethod - Whether user has a payment method on file
 * @returns true if renewal is within 30 days and no payment method
 */
function needsAttention(daysUntilRenewal: number, hasPaymentMethod: boolean): boolean {
    return daysUntilRenewal <= 30 && !hasPaymentMethod;
}

/**
 * Processes raw user data into display-ready format with computed flags.
 * @param users - Raw user records from API
 * @returns Processed users with computed fields
 */
function processUserData(users: RawUser[]): UserWithFlags[] {
    return users.map(user => {
        const daysUntilRenewal = calculateDaysUntilRenewal(user.renewal);
        return {
            id: user.id,
            name: user.name,
            isPremium: isPremiumUser(user.subscription),
            daysUntilRenewal,
            needsAttention: needsAttention(daysUntilRenewal, user.hasPaymentMethod)
        };
    });
}

// Component now focuses only on UI concerns
function UserDashboard({ userIds }: { userIds: string[] }) {
    const [users, setUsers] = useState<UserWithFlags[]>([]);
    const [loading, setLoading] = useState(true);
    const [error, setError] = useState<string | null>(null);

    useEffect(() => {
        async function loadUsers() {
            setLoading(true);
            setError(null);
            try {
                const results = await Promise.all(
                    userIds.map(id => fetch(`/api/users/${id}`).then(r => r.json()))
                );
                setUsers(processUserData(results));
            } catch (err) {
                setError(err instanceof Error ? err.message : 'Failed to load');
            } finally {
                setLoading(false);
            }
        }
        
        loadUsers();
    }, [userIds]);

    if (loading) return <Spinner />;
    if (error) return <ErrorMessage message={error} />;
    
    return (
        <div>
            {users.map(user => (
                <UserCard key={user.id} user={user} />
            ))}
        </div>
    );
}

The extracted functions (isPremiumUser, calculateDaysUntilRenewal, needsAttention, processUserData) are now completely testable without any React testing infrastructure. You can write simple unit tests:

                subscription: 'free',

        expect(result[0].needsAttention).toBe(false);
        expect(result[1].isPremium).toBe(false);
        expect(result[1].needsAttention).toBe(true);
    });
});

These tests run in milliseconds and provide confidence that the business logic is correct. You can run them on every commit, giving you immediate feedback when logic changes break expected behavior.

Case Study: E-Commerce Order Processing

A team building an e-commerce platform used vibe coding to generate order processing logic. Their initial AI-generated code calculated discounts, applied tax, and generated confirmation emails all in a single function nested inside a checkout component.

When they attempted to add unit tests, they realized they could not test the discount calculation without rendering the entire checkout flow, including payment processing. This made tests slow and flaky.

After refactoring using the extraction pattern described above, they separated discount calculation, tax computation, and email generation into pure functions. The test suite went from 3 integration tests taking 45 seconds to run to 47 unit tests taking 400ms. More importantly, they caught a critical bug in their discount logic during development that would have resulted in incorrect pricing for bulk orders.

4.3.3 Explicit Error Handling

AI-generated code often assumes the happy path. It handles expected inputs but may not anticipate malformed data, network failures, or race conditions. Making edge cases explicit transforms implicit failures into testable, handleable scenarios.

Why This Matters: Unhandled errors crash your application and create poor user experiences. When your code fails in production, you need to know why and handle it gracefully. Explicit error handling turns mysterious crashes into logged, handleable events.

Consider an AI-generated function that fetches and processes data:

Before: Implicit Error Handling
async function fetchUserOrders(userId: string) {
    const response = await fetch(`/api/users/${userId}/orders`);
    const data = await response.json();
    return data.orders;
}

This code will throw if the network fails, if the user does not exist, if the response is not valid JSON, or if the response structure differs from expectations. The caller has no idea what could go wrong. Here is the transformed version:

After: Explicit Error Handling
class UserOrdersError extends Error {
    constructor(
        message: string,
        public readonly code: string,
        public readonly userId: string,
        public readonly statusCode?: number
    ) {
        super(message);
        this.name = 'UserOrdersError';
    }
}

interface Order {
    id: string;
    total: number;
    status: string;
    items: Array<{ productId: string; quantity: number }>;
}

type FetchResult<T> = 
    | { success: true; data: T }
    | { success: false; error: UserOrdersError };

/**
 * Fetches orders for a specific user.
 * @param userId - The user ID to fetch orders for
 * @returns Result containing orders or error details
 */
async function fetchUserOrders(userId: string): Promise<FetchResult<Order[]>> {
    if (!userId || userId.trim() === '') {
        return { 
            success: false, 
            error: new UserOrdersError('User ID is required', 'INVALID_INPUT', userId) 
        };
    }

    try {
        const response = await fetch(`/api/users/${userId}/orders`);
        
        if (response.status === 404) {
            return {
                success: false,
                error: new UserOrdersError(
                    `User ${userId} not found`,
                    'USER_NOT_FOUND',
                    userId,
                    404
                )
            };
        }
        
        if (!response.ok) {
            return {
                success: false,
                error: new UserOrdersError(
                    `API returned ${response.status}`,
                    'API_ERROR',
                    userId,
                    response.status
                )
            };
        }
        
        const data = await response.json();
        
        if (!data.orders || !Array.isArray(data.orders)) {
            return {
                success: false,
                error: new UserOrdersError(
                    'Invalid response format',
                    'INVALID_RESPONSE',
                    userId
                )
            };
        }
        
        return { success: true, data: data.orders };
    } catch (error) {
        const message = error instanceof Error ? error.message : 'Unknown error';
        return {
            success: false,
            error: new UserOrdersError(message, 'NETWORK_ERROR', userId)
        };
    }
}

// Usage with explicit error handling
async function displayUserOrders(userId: string) {
    const result = await fetchUserOrders(userId);
    
    if (!result.success) {
        switch (result.error.code) {
            case 'INVALID_INPUT':
                console.error('Please provide a valid user ID');
                return;
            case 'USER_NOT_FOUND':
                console.error('User not found. They may have been deleted.');
                return;
            case 'NETWORK_ERROR':
                console.error('Network error. Please check your connection.');
                return;
            default:
                console.error('Failed to load orders:', result.error.message);
                return;
        }
    }
    
    const orders = result.data;
    console.log(`Found ${orders.length} orders for user ${userId}`);
    // Render orders...
}

The result type makes all possible outcomes explicit. Callers know exactly what to handle. Each error type has a code that enables conditional handling. The error class carries context (userId, statusCode) that helps with debugging. You can write tests for each error condition:

Testing Error Handling

    it('returns error for empty userId', async () => {
        const result = await fetchUserOrders('');
        expect(result.success).toBe(false);
        expect(result.error.code).toBe('INVALID_INPUT');
    });
    
    it('returns error for non-existent user', async () => {
        // Mock fetch to return 404

            status: 404,
            ok: false
        } as Response);

        expect(result.success).toBe(false);
        expect(result.error.code).toBe('USER_NOT_FOUND');

4.3.4 Documentation: Contracts for Humans and AI

Document interfaces with JSDoc comments so that future developers (and AI) can understand the contract each function fulfills. This documentation serves multiple purposes: it helps human developers understand intent, enables IDE autocomplete, and provides context for AI assistants that continue working on the codebase.

Why This Matters: Code is read far more often than it is written. Good documentation accelerates debugging, refactoring, and feature development. It also helps AI assistants generate better code because they have clear specifications to work with.

Effective documentation goes beyond repeating what the code does. It explains why the function exists, what assumptions it makes, and what behavior callers should expect:

Comprehensive JSDoc Documentation
/**
 * Calculates the total price for a shopping cart with applicable discounts and taxes.
 * 
 * This function handles the complete pricing calculation pipeline:
 * 1. Sums item totals
 * 2. Applies bulk discounts (see pricing rules)
 * 3. Adds tax based on shipping destination
 * 4. Applies promotional codes if valid
 * 
 * @param cart - The shopping cart with items and quantities
 * @param shippingAddress - Used for tax calculation (state determines tax rate)
 * @param promoCode - Optional promotional code for discounts
 * @returns The final calculated price in cents
 * @throws {InvalidCartError} If cart is empty or has invalid items
 * @throws {InvalidAddressError} If shipping address is incomplete
 * 
 * @example
 * const cart = { items: [{ productId: 'SKU123', quantity: 2, price: 1999 }] };
 * const address = { state: 'CA', country: 'US' };
 * const total = calculateCartTotal(cart, address, 'SAVE20');
 * // Returns 3598 (2 * 1999 + tax - 20% discount)
 * 
 * @pricingRules
 * - Bulk discount: 10% off for 5+ items, 15% off for 10+ items
 * - Tax rates: CA=8.75%, NY=8.0%, TX=6.25%, other=5.0%
 * - Promo codes: SAVE20=20% off, FREESHIP=free shipping (handled separately)
 */
function calculateCartTotal(
    cart: Cart,
    shippingAddress: Address,
    promoCode?: string
): number {
    // Implementation...
}

Notice the documentation includes:

  • What the function does. Clear description of the calculation pipeline.
  • Input parameters. What each parameter represents and its purpose.
  • Return value. The type and units of the return value (cents, not dollars).
  • Thrown errors. What can go wrong and under what conditions.
  • Examples. Concrete usage showing expected inputs and outputs.
  • Business rules. Domain-specific information that affects behavior.

4.3.5 Integration: Connecting Extracted Functions

Once you have extracted pure functions with proper typing and error handling, the final step is connecting them into your application. The component layer should be thin, focusing on user interaction while delegating business logic to the extracted functions.

This pattern relates to concepts in other chapters. Section 4.4 covers testing and iteration loops, where the testable functions you create here become the foundation for comprehensive test coverage. The state management patterns from Section 4.2 work better with pure functions because you can update state based on predictable function outputs. The AI prompting strategies in Chapter 3 become more effective when your codebase has clear interfaces that AI can understand and extend.

Skill Demonstration: The Complete Refactoring Workflow

Here is the step-by-step workflow for transforming AI-generated code into production-ready, testable code:

  1. Run the code. Verify the AI-generated code works for the happy path before refactoring.
  2. Identify boundaries. Find where data enters your application (API calls, form inputs, configuration).
  3. Add types. Define interfaces for all data structures at these boundaries.
  4. Extract pure functions. Pull business logic out of components into testable functions.
  5. Add error handling. Make failure cases explicit with result types or thrown errors.
  6. Document. Add JSDoc comments explaining intent, not just implementation.
  7. Write tests. Create unit tests for the extracted functions.
  8. Verify. Run tests and verify the refactored code produces the same results.

This workflow transforms the vibe coding output into code that is maintainable, testable, and ready for production deployment.
