2.6 Synergy in Action: Mini Example

Let's walk through building a simple RAG (Retrieval-Augmented Generation) assistant using the Synergy Triangle. This concrete example shows how all three pillars work together to transform an idea into a shippable product. We'll use a realistic scenario: building an internal chatbot that answers questions about company documentation.

The Scenario

Imagine you're at a mid-sized company with thousands of pages of internal documentation: HR policies, engineering guides, product specs, onboarding materials. Employees constantly ask the same questions, and the support team is overwhelmed. The idea is to build an AI chatbot that answers these questions by reading the documentation.

This is a classic RAG use case: the AI retrieves relevant documents, then generates a response based on that content. But the path from concept to shipped product is where the Synergy Triangle matters.

Step 1: Vibe Coding - Rapid Exploration

Who: PM + Developer working together

The PM describes what they want: "A chatbot that answers questions about our company docs. Users type a question, the bot finds the relevant information, and gives a clear answer with sources."

The developer uses vibe coding to build a prototype in hours. They use Cursor to generate the basic structure: a frontend for typing questions, a retrieval system that searches the documents, a prompt that instructs the AI to answer based on retrieved content, and a display that shows the answer with citations.

The prototype isn't polished - it has basic UI, the retrieval might be crude, and the formatting is simple. But it works. The PM can type a question and get an answer. They can test whether the concept is viable before investing more.

The Vibe Coding Session

Developer to AI: "Create a React component for a Q&A interface. Left side has an input where users type questions. Right side shows the answer. Below the answer, show the sources that were used. Use a clean, professional style." This takes 30 seconds. Then: "Add a simple retrieval function that takes a query, embeds it, searches a vector store, and returns the top 5 chunks." This takes another 30 seconds. In minutes, the skeleton is there.

Fun Note: The 30-Second Build

The first compiled programming language, FORTRAN, took months to write and could perform about 100 calculations per second. Today, AI generates working React components in seconds that run millions of calculations per second on modern hardware. The tools change but human creativity remains the constant multiplier.

Step 2: Define Evals - Establishing Success Criteria

Who: PM owns, with engineering input

Before going further, the PM needs to define what success looks like. They create a golden dataset: 50 Q&A pairs covering the most common questions. These aren't synthetic examples - they're real questions employees ask, with answers that subject matter experts have verified as correct.

The PM also defines eval criteria. For this product, success means:

  • Factual accuracy: The answer must be correct according to the source documents
  • Source citation: The answer must cite the documents it used
  • Coverage: If the question can't be answered from the docs, the system should say so
  • Clarity: The answer should be understandable and well-organized

These criteria become the eval suite. Every change to the system is tested against these 50 examples. If a change improves eval scores, it's good. If it hurts eval scores, it's bad. No more subjective debates about whether the product is getting better.

Step 3: Engineering Optimization - Systematic Improvement

Who: Engineering leads, guided by evals

With the evals in place, the engineering team systematically improves the system. They experiment with:

Chunk sizes: Should documents be split into 500-word chunks or 1000-word chunks? They test both against evals.

Embedding models: Which embedding model retrieves the most relevant documents? They test several.

Retrieval parameters: How many chunks should they retrieve? 3? 5? 10? They test each against evals.

Prompt variations: How should the AI be instructed to use the retrieved content? They test different prompt approaches.

Each experiment is measured against the eval suite. The results are clear: chunk size of 800 words with 5 retrieved chunks and a specific prompt template achieves 87% accuracy on the eval set. Other combinations achieve lower scores.

The Evidence-Based Approach

Without evals, this optimization would be guesswork. "I think the longer chunks are better." "I feel like more context helps." With evals, it's evidence. "The eval score improved from 72% to 87% when we switched to 800-word chunks." This transforms engineering productivity - every change is validated, and only improvements that actually improve the product are shipped.

Step 4: Production Hardening - Making It Shippable

Who: Engineering

The optimized prototype isn't ready for production. The engineering team adds what's needed for real-world use:

Observability: They add detailed logging so they can see what's happening in production: what queries are being asked, what documents are being retrieved, what answers are being generated. If something goes wrong, they can debug it.

Error handling: What happens if the vector database is down? What happens if the API rate limit is hit? They add fallbacks and retry logic so the system degrades gracefully.

Cost monitoring: They add tracking so they know how much each query costs. They set up alerts if costs spike unexpectedly.

Rate limiting: They add limits so one user can't overload the system with thousands of queries.

Security: They ensure the system only accesses documents the user is allowed to see. They add input sanitization to prevent prompt injection.

After this hardening, the prototype is now a shippable product. It can be released to real users in real conditions.

The Complete Cycle

This example illustrates how the Synergy Triangle creates value:

  1. Vibe coding enabled rapid exploration - they built a working prototype in hours rather than weeks
  2. Evals enabled evidence-based optimization - they knew which changes actually improved the product
  3. Production engineering enabled reliable delivery - the product works in real conditions with real users

Each pillar enabled the others. Vibe coding would have been aimless without evals to optimize toward. Evals would have been academic without engineering to make them practical. Engineering would have been inefficient without the rapid iteration that vibe coding enables.

The Synergy Principle

Each pillar enables the others. Vibe coding makes PM hypothesis testing fast - you can try many directions quickly. Evals make engineering optimization meaningful - you can tell which changes improve the product. Engineering makes prototypes shippable - what was a proof-of-concept becomes a real product. The triangle isn't just a framework for understanding - it's a practical guide for how to work.

Chapter 2 Summary: The Synergy Triangle

This chapter introduced the core framework that guides the rest of the book. The Synergy Triangle has three pillars:

  • Vibe Coding: High-velocity development using AI-native tools. Enables rapid hypothesis testing and fast iteration.
  • AI Product Management: Eval-driven development and clear success criteria. Prevents engineering waste and enables evidence-based decisions.
  • AI Engineering: Production infrastructure that makes prototypes shippable. Adds reliability, scalability, and observability.

These pillars connect through feedback loops: vibe coding informs PM, PM informs engineering, engineering enables more ambitious vibe coding. Together, they create a virtuous cycle that accelerates AI product development.

The anti-patterns we covered show what happens when one pillar is missing: impressive demos that fail in production, endless debates without resolution, slow processes that lose the speed advantage, and economics that become unsustainable at scale. The solution in each case is to strengthen the weak pillar and ensure all three are connected.
