17.3 Cross-Role Collaboration

AI product development requires unprecedented collaboration between roles that traditionally worked in silos. A Product Manager must understand AI capabilities, an AI Engineer must care about user experience, and a Vibe Developer must think about production constraints. This section shows you how to make this collaboration work.

Why This Matters

The #1 cause of AI product failure is not technical. It's misalignment between teams: PMs who don't understand what's possible, Engineers who don't understand what users need, and Designers who create experiences that can't be implemented. Clear role definitions and collaboration patterns prevent this miscommunication.

Fun Note: The Emoji That Solved the Problem

A PM and Engineer argued for 2 weeks about how to handle "ambiguous user intent." During a heated debate, the Engineer sent an emoji of a shrug. Everyone laughed. Then they realized the answer was literally to show users an "I'm not sure what you mean" message. The $50,000 architectural debate was solved by a Unicode character.

The Trust Architecture: What PMs Define

The PM owns defining "trust architecture" - what makes users trust the AI's output. This isn't just about accuracy; it's about the entire experience of reliability.

Analogy: Trust Architecture Is Like a Restaurant's Reputation

Think of trust architecture like a restaurant. You don't just trust a restaurant because the food is accurate (correct temperature, correct ingredients). You trust it because the menu is clear, the staff explains what they recommend, they admit when they're out of something, and they fix mistakes promptly. AI trust works the same way - accuracy matters, but transparency, acknowledgment of uncertainty, and clear escalation paths matter just as much.

Skill Demonstration: Defining Trust Architecture

For a medical AI assistant, the PM defined these trust elements:

  • Source transparency: Every claim cites a source document
  • Uncertainty acknowledgment: AI says "I'm not certain" when appropriate
  • Escalation path: Clear button to connect with human doctor
  • Scope clear: Explicit statement of what's out of scope
  • Traceability: Users can see the reasoning path

What the PM Actually Owns

  • Problem definition: What user pain point are we solving?
  • Success metrics: How do we know it's working?
  • Quality criteria: What does "good enough" look like?
  • Risk assessment: What could go wrong and what's the impact?
  • User feedback loops: How do we hear from users?
Real-World Scenario: PM/Engineer Misalignment

A PM requested "the AI should understand context across the entire conversation." The AI Engineer implemented it, but it cost 10x more than simpler alternatives and added 2 seconds of latency. The real user need was understanding the last 3 messages, not the entire conversation. The fix required the PM to be more specific about "context" and the Engineer to ask "how much context is actually needed?"

Retrieval: What AI Engineers Own

The AI Engineer owns retrieval - the hardest problem in AI products. Even the best language model is only as good as what it's retrieving.

Why Retrieval Is Hard

Consider a legal document search: "find the clause about liability for data breaches" should match "indemnification for security incidents" even though the words don't match. Users ask in their language; documents are written in legal language. Bridging this gap is genuinely difficult.

What the AI Engineer Actually Owns

  • Model selection: Which model and why?
  • Retrieval pipeline: How do we find the right content?
  • Prompt engineering: How do we get the best outputs?
  • Cost optimization: How do we keep costs reasonable?
  • Quality metrics: How do we measure retrieval accuracy?
Code Example: Retrieval Quality Ladder
// The AI Engineer's toolkit for improving retrieval:

// Level 1: Simple keyword matching
const results = await search(query); // BM25/elasticsearch

// Level 2: Add semantic search
const embeddings = await embed(query);
const semanticResults = await vectorSearch(embeddings);

// Level 3: Hybrid (keyword + semantic)
const combinedResults = await hybridSearch(query, embeddings);

// Level 4: Re-ranking
const reranked = await rerank(query, combinedResults);

// Level 5: Query understanding/transformation
const transformed = await rewriteQuery(query);
const finalResults = await hybridSearch(transformed);

// Each level adds cost and latency - choose based on user need

Rapid Iteration: What Vibe Developers Own

Vibe Developers accelerate iteration by shipping features in hours or days, not weeks. They prototype quickly and know when to refactor for production.

What the Vibe Developer Actually Owns

  • Rapid prototyping: Get something working fast
  • User experience: Make it feel good to use
  • Frontend/UI: The interface users interact with
  • Integration: Connect AI to existing systems
  • Production hardening: Refactor prototype to production-ready
Skill Demonstration: Vibe to Production Pipeline

A Vibe Developer used Cursor to build a prototype in 2 days. Then, over the next 2 weeks, they systematically refactored it:

  • Day 1-2: Working prototype with hardcoded mock data
  • Day 3: Replace mocks with real API calls
  • Day 4: Add error handling and loading states
  • Day 5: Add caching for repeated queries
  • Week 2: Add observability, rate limiting, and testing

The key: they didn't try to make the prototype production-ready. They accepted it would be thrown away and focused on learning quickly.

Collaboration Patterns That Work

These collaboration patterns have proven effective across many AI product teams:

1. Weekly Sync: PM + AI Engineer + Vibe Developer

Every week, these three meet to align on:

  • What's working (show metrics)
  • What's blocked (raise blockers)
  • What's next (prioritize backlog)
Real-World Scenario: Weekly Sync That Caught a Problem

During weekly sync, the Vibe Developer mentioned that the new feature "felt slower" but they didn't have metrics yet. The PM pushed to add latency tracking that week. Two weeks later, they caught a regression that was adding 500ms - before it would have become a user complaint.

2. Eval Review: Everyone Reviews Quality Together

Once a week, the entire product team reviews sample outputs together. This keeps everyone aligned on quality standards. 

Eval Review Meeting Agenda:
1. Review 10 random outputs from production (15 min)
2. Discuss: Is this good enough? Why or why not? (15 min)
3. Identify patterns: What's working? What's not? (10 min)
4. Action items: What needs to change? (10 min)

3. Design Review: PM Reviews UX Before Implementation

Before building, the PM reviews the Vibe Developer's UI/UX designs to ensure they align with trust architecture requirements (citations, uncertainty, etc.).

4. Technical Review: AI Engineer Reviews Vibe Developer's Production Code

Before shipping, the AI Engineer reviews the Vibe Developer's code for cost optimization, security, and scalability concerns.

Escalation Paths

Know when to escalate and to whom:

Can't ship fast enoughVibe Developer"How do we prototype this faster?"
Quality not good enoughWhole Team"Eval score dropped - what do we do?"
Key Takeaways
  • PM owns trust architecture - what makes users trust the AI
  • AI Engineer owns retrieval - the hardest technical problem
  • Vibe Developer owns rapid iteration and UX
  • Weekly syncs prevent misalignment
  • Regular eval reviews keep everyone focused on quality
