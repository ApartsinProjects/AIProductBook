17.2 Templates and Checklists

Templates and checklists transform tribal knowledge into repeatable processes. They ensure nothing falls through the cracks and make it easy to onboard new team members. Use these as starting points that you adapt to your specific context.

Why This Matters

AI product development involves many moving parts. Without structured checklists, teams frequently forget critical steps like adding rate limiting or documenting fallback behavior. Templates accelerate onboarding and ensure consistent quality across the team.

Analogy: The Pre-Flight Checklist

Pilots don't rely on memory for critical steps - they use checklists. Before every flight, they systematically go through dozens of items, not because they're forgetful, but because the stakes are too high to miss anything. AI product development works the same way. Your "flight" is shipping to production, and the checklist ensures you don't forget the critical steps that prevent disasters.

Fun Note: The Checklist That Caught the Uncatchable

A team added "test on mobile" to their checklist as an afterthought. During that test, they discovered their AI feature was completely unusable on mobile due to a context length issue that would have affected 40% of users. The "obvious" test nobody had time for caught a critical bug.

Eval Creation Checklist

Before writing your first line of evaluation code, work through this checklist to ensure you have the right foundation. 

Eval Creation Checklist:

PHASE 1: Define Success Metrics
□ Identify the primary business outcome your AI impacts
□ Define user-facing quality dimensions (accuracy, speed, helpfulness)
□ Set target performance thresholds for each dimension
□ Establish how you'll measure improvement over time

PHASE 2: Create Golden Dataset
□ Collect 50+ representative examples (more for complex products)
□ Include edge cases and failure modes (at least 20%)
□ Ensure diversity in query types, lengths, and complexity
□ Document the expected output for each example
□ Have 3+ humans annotate each example for inter-annotator agreement

PHASE 3: Write Evaluation Prompts
□ Define rubric with clear scoring criteria (0-5 scale works well)
□ Write instructions that leave no room for interpretation
□ Test with human raters before automated eval
□ Include edge case handling in the rubric

PHASE 4: Establish Baseline
□ Run baseline eval on current/golden standard version
□ Document baseline scores for each quality dimension
□ Calculate inter-rater reliability (aim for 80%+ agreement)
□ Create dashboard to track scores over time
Skill Demonstration: Creating a Golden Dataset

For a code review AI assistant, a properly constructed golden dataset included:

  • 100 real GitHub PR descriptions from users
  • 20 examples of "should skip" (PRs needing human review only)
  • 30 edge cases (very long PRs, ambiguous descriptions, multi-file changes)
  • Each annotated with: expected action, confidence level, explanation

This dataset revealed that the initial model struggled with PRs over 500 lines, leading to adding a pre-processing step to summarize long diffs.

Production Readiness Checklist

Before shipping to any user (even alpha), ensure you've addressed these production concerns. 

Production Readiness Checklist:

EVAL-DRIVEN DEVELOPMENT
□ Eval runs automatically on every code change
□ Eval scores trend is visible on dashboard
□ Threshold for blocking deployment is defined
□ Regression detection is automated

COST OPTIMIZATION
□ Per-request cost is tracked and capped
□ Model routing based on complexity is implemented
□ Caching strategy is defined (what can be cached?)
□ Fallback to cheaper model on failure works
□ Budget alerts configured for unexpected usage spikes

OBSERVABILITY
□ Tracing is implemented for all AI calls (input/output/timing)
□ Metrics dashboard shows latency, cost, quality
□ Error logging includes sufficient context for debugging
□ Health check endpoint exists for monitoring

ERROR HANDLING AND FALLBACKS
□ Timeout handling (max wait time before fallback)
□ API failure recovery (retry with exponential backoff)
□ Graceful degradation (what happens if AI is unavailable?)
□ User-facing error messages are helpful, not technical

SECURITY REVIEW
□ Input validation blocks prompt injection attempts
□ Output filtering prevents sensitive data leakage
□ Authentication and authorization verified
□ Rate limiting is implemented per user/IP
□ PII handling complies with privacy requirements

LOAD TESTING
□ Tested at 2x expected peak traffic
□ Behavior under sustained load verified
□ Resource limits identified and configured
□ Cascading failure scenarios tested

ROLLBACK PLAN
□ Previous version can be deployed in under 5 minutes
□ Rollback triggers are clearly defined
□ Database migrations can be reversed
□ Rollback communication plan exists
Real-World Scenario: The Checklist That Saved Them

A startup rushed to ship an AI feature before a conference. They skipped the security review checklist item about input validation. Within 24 hours of launch, users had discovered they could inject arbitrary prompts through the chat interface, causing the AI to reveal system instructions. The fix required an emergency deployment and a 3-day incident response. The lesson: the checklist exists for a reason.

Use this template when proposing a new AI feature or product.

[One paragraph describing the specific user problem this AI feature solves]

- How does their workflow change?
- What's the before/after?

## 3. Success Metrics
| Metric | Target | How Measured |
|--------|--------|--------------|

| Secondary: [e.g., User satisfaction] | [e.g., 4.5/5] | [e.g., Survey] |

## 4. AI Approach
- Model selection rationale
- Retrieval strategy (if applicable)
- Input/output constraints

## 5. Eval Strategy
- Golden dataset size: [e.g., 100 examples]
- Key quality dimensions: [e.g., accuracy, relevance, tone]
- Baseline target: [e.g., 80% on eval]

## 6. Cost Analysis
- Expected queries per day: [e.g., 10,000]
- Estimated cost per 1K queries: [e.g., $0.50]
- Monthly cost estimate: [e.g., $1,500]

## 7. Guardrails
- Input validation: [describe]
- Output filtering: [describe]
- Rate limiting: [describe]

## 8. Rollout Plan
- Phase 1: [e.g., 5% of users, 2 weeks]
- Phase 2: [e.g., 25% of users, 2 weeks]
- Phase 3: [e.g., 100%]

## 9. Kill Conditions
- [e.g., Eval score drops below 70%]
- [e.g., Cost exceeds $10K/month]
- [e.g., Critical security issue found]

Key Takeaways
  • Checklists prevent critical oversights that cause incidents
  • Templates accelerate team workflow and ensure consistency
  • Customize templates to your context, but start with proven ones
  • Review and update templates after each major incident
