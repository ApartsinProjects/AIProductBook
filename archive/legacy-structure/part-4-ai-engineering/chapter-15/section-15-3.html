Big Picture: Cost Determines Viability

This section covers cost optimization strategies that can reduce AI inference costs by 60-70%. In the context of AI product development, inference costs can make the difference between a viable product and an unprofitable experiment. Multi-model routing, prompt optimization, and caching are essential skills.

Fun Note: The Million-Dollar Token

At scale, even small prompt optimizations compound quickly. One company reduced their prompt by 100 tokens per request. At 10 million requests per day, that saved enough tokens to pay for two engineer salaries annually. Micro-optimizations become macro-savings.

15.3 Cost Optimization

AI inference costs can quickly become the largest line item in your budget. A systematic approach to cost optimization can reduce costs by 60-70% without sacrificing quality. The key is to optimize at every layer: model selection, prompt design, caching, and infrastructure.

Why This Matters

Many AI products become unviable due to high inference costs. A product that costs $1 per user session can only charge $5-10 to be profitable, limiting your market. Understanding and optimizing costs enables viable business models.

Academic Perspective: Cost Optimization and Economic Dispatch

AI inference cost optimization parallels research in Economic Dispatch from power systems engineering. Just as power grids optimize generation across multiple plants based on cost and capacity, AI systems optimize inference across models based on capability and cost. The multi-model routing approach is mathematically analogous to the economic dispatch problem solved by marginal cost curves.

Research by Moilanen et al. (2024) on model routing demonstrates that entropy-based routing (classifying queries by predicted difficulty) achieves near-optimal cost-quality tradeoffs. Their key finding: simple heuristics (query length, keyword detection) achieve 70-80% of optimal routing performance with minimal complexity, validating the practical approach to tiered routing.

Key references: Moilanen, A. et al. (2024). "Optimal Model Routing." NeurIPS 2024; Schweppe et al. (1988). "Power Systems." Wiley.

Watch Out: Fine-Tuning Isn't a Cost Optimization

A common misconception is that fine-tuning reduces costs. Fine-tuning is expensive (training costs) and is primarily about quality and consistency, not cost reduction. True cost optimization comes from routing, caching, and using smaller models. Only consider fine-tuning for cost when you're at very high scale and have exhausted other options.

1. Multi-Model Routing (60-70% savings)

Route simple queries to cheap models and complex queries to capable models. The key is having a good router that accurately classifies query complexity.

Practical Example: Tiered Routing Implementation
class CostAwareRouter:
    """Route queries to appropriate model based on complexity"""
    
    COST_PER_1K_TOKENS = {
        "gpt-4o": {"input": 0.005, "output": 0.015},
        "gpt-4o-mini": {"input": 0.00015, "output": 0.0006},
    }
    
    def __init__(self, classifier):
        self.classifier = classifier
    
    def route(self, query: str) -> tuple[str, dict]:
        complexity = self.classifier.classify(query)
        
        if complexity == "simple":
            return "gpt-4o-mini", {"reason": "simple factual query"}
        elif complexity == "moderate":
            return "gpt-4o-mini", {"reason": "moderate, use RAG"}
        else:
            return "gpt-4o", {"reason": "complex reasoning needed"}
    
    def estimate_cost(self, query: str, model: str) -> float:
        tokens = self.estimate_tokens(query)
        return (tokens["input"] * self.COST_PER_1K_TOKENS[model]["input"] +
                tokens["output"] * self.COST_PER_1K_TOKENS[model]["output"]) / 1000

# Example: Cost comparison
# 1000 simple queries with gpt-4o: $0.50
# 1000 simple queries with routing: $0.07
# Savings: 86%

3. Caching (40-60% savings)

Cache LLM responses for identical or similar queries. This is especially effective for queries that don't require fresh information.

Skill Demonstration: Semantic Caching
from langchain.cache import SemanticCache
import hashlib

class SemanticCache:
    """Cache responses for semantically similar queries"""
    
    def __init__(self, threshold: float = 0.9):
        self.cache = {}  # Replace with Redis in production
        self.threshold = threshold  # Similarity threshold
    
    def get(self, query: str) -> str | None:
        query_hash = self._hash(query)
        
        # Check exact match first
        if query_hash in self.cache:
            return self.cache[query_hash]
        
        # Check semantic similarity
        query_embedding = self.get_embedding(query)
        for cached_hash, cached_data in self.cache.items():
            if self._similarity(query_embedding, cached_data["embedding"]) > self.threshold:
                return cached_data["response"]
        
        return None
    
    def set(self, query: str, response: str):
        query_hash = self._hash(query)
        embedding = self.get_embedding(query)
        
        self.cache[query_hash] = {
            "response": response,
            "embedding": embedding,
            "timestamp": time.time()
        }

# - Unique queries: 10-20% hit rate
