14.2 Fine-tuning vs Distillation

Objective: Choose and improve models wisely with the right evaluation frameworks.

Prerequisites: Completion of Section 14.1 recommended.
Big Picture: Customization Tradeoffs

This section covers the decision between fine-tuning, distillation, and model merging for customizing AI behavior. In the context of AI engineering, these techniques enable you to balance capability, cost, and control. Understanding when each approach makes sense prevents costly customization mistakes.

Fun Note: The $500k Fine-Tuning Mistake

A healthcare company spent $500,000 fine-tuning a model for medical coding, only to discover that better prompting achieved 95% of the same results at 1% of the cost. The lesson: always exhaust prompt engineering before resorting to fine-tuning.

14.2 Fine-tuning vs Distillation

Customization options exist on a spectrum from simple prompt optimization to full model fine-tuning. Understanding when each approach makes sense is essential for making cost-effective engineering decisions.

Why This Matters

Fine-tuning is expensive and changes are hard to reverse. Distillation can give you a smaller, cheaper model but may lose capability. Making the wrong choice wastes significant development time and money.

Watch Out: Fine-Tuning Is Irreversible (Mostly)

Once you fine-tune a model, you cannot "un-tune" it. If you fine-tune on bad data or with wrong parameters, the only solution is to re-fine-tune from the base model. LoRA adapters help by keeping base model weights intact, but the fine-tuned behavior is still hard to reverse. Always test extensively on held-out data before committing to a fine-tuned model in production.

Option 1: Fine-tuning

Fine-tuning adapts a pre-trained model to your specific data and requirements. This is appropriate when you need consistent behavior that can't be achieved through prompting alone.

Case Study: Legal Document Processor

A legal tech company needed consistent extraction of specific clauses from contracts. They fine-tuned a 7B model on 500 annotated examples:

# Fine-tuning setup with LoRA
from unsloth import FastLanguageModel
from trl import SFTTrainer
from transformers import TrainingArguments

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="meta-llama/Llama-3.2-7B-Instruct",
    max_seq_length=2048,
    dtype=None,
    load_in_4bit=True,
)

# Add LoRA adapters for efficient fine-tuning
model = FastLanguageModel.get_peft_model(
    model,
    r=16,
    lora_alpha=16,
    target_modules=["q_proj", "v_proj"],
    lora_dropout=0,
)

# Train on contract data
trainer = SFTTrainer(
    model=model,
    tokenizer=tokenizer,
    train_dataset=contract_dataset,
    dataset_text_field="text",
    max_seq_length=2048,
    args=TrainingArguments(
        per_device_train_batch_size=2,
        gradient_accumulation_steps=4,
        warmup_steps=10,
        num_train_epochs=3,
        learning_rate=2e-4,
    ),
)
trainer.train()

After fine-tuning, the model consistently extracted clause types with 92% accuracy vs 70% with prompting alone.

When Fine-tuning is Appropriate

  • Consistent output format required across thousands of queries
  • Domain-specific reasoning patterns needed
  • Cost optimization for high-volume inference
  • Clear, stable requirements that won't change frequently
Watch Out: Fine-Tuning vs RAG Knowledge

Fine-tuning does not inject knowledge. If you need the model to know specific facts (like your company's return policy), fine-tuning will not help unless those facts appear in your training data. For knowledge injection, use RAG. Fine-tuning is about how to respond, not what to respond with.

Option 2: Distillation

Distillation trains a smaller model (student) to mimic a larger model (teacher). The student learns to approximate the teacher's outputs, preserving most capability at lower cost.

Practical Example: Creating an SLM
from distilbert import DistilBertForQuestionAnswering
import torch

# Teacher: Large model that generates outputs
teacher_model = load_pretrained("gpt-4o")

# Student: Smaller model to train
student_model = DistilBertForQuestionAnswering.from_pretrained(
    "distilbert-base-uncased"
)

# Distillation training
def distillation_loss(student_logits, teacher_logits, temperature=2.0):
    """Knowledge distillation loss"""
    # Soft targets from teacher (with temperature)
    soft_teacher = F.softmax(teacher_logits / temperature, dim=-1)
    soft_student = F.log_softmax(student_logits / temperature, dim=-1)
    
    # KL divergence loss
    loss = F.kl_div(soft_student, soft_teacher, reduction='batchmean')
    return loss * (temperature ** 2)

# Train student to mimic teacher
for batch in training_data:
    # Get teacher predictions
    with torch.no_grad():
        teacher_output = teacher_model(batch.input_ids)
    
    # Get student predictions
    student_output = student_model(batch.input_ids)
    
    # Compute distillation loss
    loss = distillation_loss(
        student_output.logits,
        teacher_output.logits
    )
    
    loss.backward()
    optimizer.step()

After distillation, the student model (3B params) retained 85% of the teacher (70B params) capability while being 10x faster and 20x cheaper to run.

Option 3: Model Merging

Model merging combines weights from multiple specialized models without additional training. This can create models that perform well across multiple tasks.

Real-World Scenario: Merging Specialized Models

A team had three fine-tuned models:

  • Model A: Great at code generation
  • Model B: Excellent at math reasoning
  • Model C: Best at creative writing

They merged them using TIES merging:

from mergekit import MergeConfiguration

config = MergeConfiguration(
    models=[
        {"model": "code-model", "weight": 1.0},
        {"model": "math-model", "weight": 1.0},
        {"model": "writing-model", "weight": 1.0},
    ],
    merge_method="ties",
    normalize=True,
)

merged_model = mergekit.merge(config)

The merged model performed at 90% of each specialized model in their domain while being a single unified model.

ApproachWhen to UseCostTradeoffs
Fine-tuningSpecific output style, domain patternsHighExpensive to train, changes hard to reverse
DistillationCreate efficient SLMs from large modelsMediumMay lose some capability
Model mergingCombine specialized modelsLowLess predictable results
Start with Prompting

Before investing in fine-tuning, exhaust prompt engineering options. Most behavior can be achieved through clever prompting at a fraction of the cost. Only fine-tune when you have clear evidence that prompting can't achieve your quality bar.
