14.1 Model Router Strategies

Objective: Choose and improve models wisely with the right evaluation frameworks.

Prerequisites: Completion of Chapters 11-13 recommended.
Big Picture: Use the Right Model for the Task

This section covers model routing strategies that can reduce AI costs by 60-70% while maintaining quality. In the context of AI product development, routing is essential for viable economics. The difference between routing everything to GPT-4o and using intelligent routing is often the difference between a profitable product and an expensive experiment.

Fun Note: The Router That Pays For Itself

A team at a fintech startup discovered their routing system paid for itself in one week. By routing 80% of simple queries to a cheaper model, they saved $12,000/month in API costs while actually improving response times. The routing classifier cost $50/month to run.

14.1 Model Router Strategies

Not all queries require the most capable model. A well-designed routing system sends simple tasks to cheaper, faster models while reserving expensive models for complex tasks that need their capabilities. This approach can reduce costs by 60-70% while maintaining quality.

Why This Matters

The difference in cost between models can be 10-20x. Using GPT-4o for a simple "What is 2+2?" question is wasteful. A router that classifies query complexity and routes appropriately lets you get the best quality per dollar.

Watch Out: Routing Mistakes Cost More Than Routing

A common worry is that routing adds cost. The reality is that routing adds minimal cost (a few milliseconds and a tiny token count) while saving significant money through model routing. The cost of a routing classifier is far less than the cost of over-routing to expensive models. The risk is under-routing (sending complex queries to cheap models that fail), which creates poor user experience. When in doubt, route up.

The Routing Principle

Use the right model for the right task. This is not about using the cheapest model possible, but about matching model capability to task complexity.

Implementation Approaches

1. Heuristic Routing

Use simple rules based on query characteristics like length, keywords, or explicit indicators.

Practical Example: Heuristic Router
def route_query(query: str) -> str:
    """Simple heuristic routing"""
    query_lower = query.lower()
    
    # Complex indicators - route to capable model
    complex_indicators = [
        "compare", "analyze", "debug", "explain why",
        "troubleshoot", "design", "write code for"
    ]
    
    for indicator in complex_indicators:
        if indicator in query_lower:
            return "gpt-4o"
    
    # Simple indicators - cheap model sufficient
    simple_indicators = [
        "what is", "when does", "where is", "how to",
        "list", "what are", "define"
    ]
    
    for indicator in simple_indicators:
        if indicator in query_lower:
            return "gpt-4o-mini"
    
    # Default to medium model
    return "gpt-4o"

# Tradeoff: Simple but doesn't capture all complexity

2. LLM-Based Routing

Use a small model to classify query complexity and route accordingly.

Skill Demonstration: LLM Router
from enum import Enum
from pydantic import BaseModel

class QueryComplexity(BaseModel):
    complexity: Literal["simple", "moderate", "complex"]
    reasoning: str

def route_with_llm(query: str) -> str:
    """Use small model to classify complexity"""
    
    classification = llm_mini.with_structured_output(QueryComplexity).invoke(f"""
        Classify this query's complexity for routing purposes.
        
        - simple: Factual questions, basic lookups, simple transformations
        - moderate: Questions requiring context, multi-step reasoning, policy application
        - complex: open-ended analysis, creative tasks, debugging, multi-part problems
        
        Query: {query}
        
        Provide complexity level and brief reasoning.
    """)
    
    if classification.complexity == "simple":
        return "gpt-4o-mini"
    elif classification.complexity == "moderate":
        return "gpt-4o"
    else:
        return "gpt-4o"

# Tradeoff: More accurate but adds latency and cost per query

3. Embedding-Based Routing

Train a classifier on historical queries to predict which model to use.

Code Example: Classifier-Based Router
from sklearn.linear_model import LogisticRegression
import numpy as np

class EmbeddingRouter:
    """Train a classifier on query embeddings to predict model"""
    
    def __init__(self):
        self.classifier = LogisticRegression()
        self.embedding_model = embeddings_model
    
    def train(self, queries, correct_model_labels):
        # Convert queries to embeddings
        X = np.array([self.embedding_model.encode(q) for q in queries])
        
        # Train classifier
        self.classifier.fit(X, correct_model_labels)
    
    def predict(self, query):
        embedding = self.embedding_model.encode(query)
        return self.classifier.predict([embedding])[0]
    
    def get_confidence(self, query):
        embedding = self.embedding_model.encode(query)
        probs = self.classifier.predict_proba([embedding])[0]
        return max(probs)

# Train on historical data where you know which model worked best
Routing with Confidence

Add a confidence threshold to your router. If confidence is low, default to the more capable model. It's better to over-pay slightly than to give users a poor experience with an under-powered model.
