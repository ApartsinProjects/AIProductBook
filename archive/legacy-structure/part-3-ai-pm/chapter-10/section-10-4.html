9.4 Governance in Agentic Systems

Objective: Build trustworthy products with proper governance frameworks.

Prerequisites: Completion of Section 9.3 recommended.
Big Picture: Governance for Autonomous AI

Agentic AI systems require governance mechanisms that go far beyond passive AI features because the stakes of failure or misuse are much higher when AI can take autonomous action. Human-in-the-loop requirements, behavioral guardrails, audit trails, and capability limits are essential controls that ensure AI Agents operate within acceptable boundaries. The principle of least privilege should guide all agentic system design.

Fun Note: The Three Laws Bug

Isaac Asimov's Three Laws of Robotics were fictional. Every sci-fi robot ever built eventually violated them because real-world situations are more complex than clean rules. Agentic AI governance faces the same problem: rules work until they do not.

9.4 Governance in Agentic Systems

Agentic AI systems, which can take autonomous actions on behalf of users, require additional governance beyond what is needed for passive AI features. When an AI can book flights, send emails, make purchases, or execute trades, the stakes of failure or misuse are much higher. Governance in agentic systems must address the unique challenges of autonomous action: what actions are permitted, who is accountable when things go wrong, and how users maintain control.

Why Agentic Systems Need More Governance

Traditional AI products generate outputs that users can review before acting on. An AI writes an email, and the user decides whether to send it. An AI suggests a response, and the user chooses to accept or ignore it. In agentic systems, the AI takes action without waiting for user approval. This fundamental shift changes the risk profile and requires governance mechanisms that would be unnecessary for passive AI.

Case Study: The Email That Was Sent

A productivity app added an "auto-send" feature where the AI would send AI-generated emails after a timeout. Users loved the time savings. Then a user discovered the AI had sent an email to their boss saying "I don't think this project is worth pursuing" without the user's knowledge. The user had been in the habit of reviewing before send but was interrupted before reviewing this particular email.

The consequence: The user had to have an uncomfortable conversation with their boss explaining that AI sent the email. Trust was damaged, and the company received negative press. The feature was eventually changed to require explicit user action for all emails to external recipients.

Human-in-the-Loop Requirements

Human-in-the-loop (HITL) governance ensures that humans approve or can override AI actions, especially for high-stakes decisions. The key principle is that the level of human oversight should be proportional to the impact of the action. Low-stakes, reversible actions can be automated. High-stakes, consequential actions require human approval or easy override.

Skill: Implementing Human-in-the-Loop

Define action tiers:

  1. Tier 1 - Fully autonomous: Actions that are easily reversed and low-impact (e.g., auto-filing emails, sorting into folders)
  2. Tier 2 - Human-in-the-loop: Actions requiring confirmation before execution (e.g., sending messages, making calendar changes)
  3. Tier 3 - Human-on-the-loop: AI executes but human must approve before completion (e.g., initiating payments, making purchases)
  4. Tier 4 - Human-only: AI can suggest but only human can execute (e.g., terminating accounts, deleting data)

Implementation: At each tier, define what the human review looks like. Is it explicit approval? A simple "yes/no" button? Or is it post-execution review?

Case Study: Stripe's Agent Governance

Stripe, a payment processing company, allows AI Agents to make payments on behalf of users. Their governance approach requires explicit user permission for each action, with clear disclosure that an AI is acting. Users must set spending limits, and any transaction above the limit requires human approval. All agent actions are logged and can be viewed in the user's activity dashboard.

Why this works: Users maintain control through explicit permissions and spending limits, while still benefiting from automation for routine payments.

Behavioral Guardrails

Guardrails are constraints that prevent AI Agents from taking actions that violate policies, even if the AI would choose to do so otherwise. While Agents are designed to take autonomous action, guardrails ensure they cannot cross certain boundaries. Effective guardrails operate at multiple levels: input validation, action checking, and output filtering.

Skill: Designing Effective Guardrails

Input guardrails:

  • Validate that user requests are within the agent's permitted scope
  • Detect and block prompt injection attempts
  • Check for harmful or illegal requests

Action guardrails:

  • Verify proposed actions against allowed action list
  • Check action parameters against safety constraints
  • Enforce rate limits and quotas

Output guardrails:

  • Validate responses before returning to user
  • Filter sensitive information
  • Check for policy violations in generated content

Audit Trails for Accountability

Agentic systems must maintain complete audit trails because when something goes wrong, you need to understand what happened. An audit trail should record the original user request, the AI's reasoning, the actions taken, the results, and any human decisions in the process. This serves both technical debugging and regulatory compliance.

Case Study: The Trading Bot Incident

A quantitative trading firm ran an AI agent that executed trades. The agent made a series of unusual trades that resulted in $2M in losses. Without an audit trail, the firm could not determine whether the losses were due to a model bug, data issues, or malicious prompt injection. With a complete audit trail, they could see the exact inputs, model outputs, and trade decisions, allowing them to identify and fix the root cause.

Key insight: Audit trails are not just for compliance; they are essential for debugging and improving agentic systems.

Skill: Building Audit Trails

Each agent action should record:

  1. Timestamp: When the action occurred
  2. User identity: Who initiated the request
  3. Request details: What the user asked for
  4. Agent reasoning: How the agent interpreted and planned to fulfill the request
  5. Actions taken: What the agent did (API calls, data access, etc.)
  6. Results: What happened as a result
  7. Human decisions: Any human approvals or overrides

Store audit logs immutably and retain them according to regulatory requirements (often 7+ years for financial services).

Capability Limits and Scope Control

Every agent should have clearly defined boundaries: what it can do, what it cannot do, and what it should not do but might accidentally try. Capability limits should be explicit and enforced technically, not just documented. Users should understand what the agent can and cannot do to set appropriate expectations.

The Principle of Least Privilege

Agentic systems should have only the minimum capabilities required to fulfill their purpose. If an agent needs to read documents, it should not also have write access unless explicitly required. Start restrictive and expand as needed.

Governance Framework Implementation

Implementing governance for agentic systems requires a structured approach:

  1. Capability mapping: Document everything the agent can do
  2. Risk assessment: For each capability, assess what could go wrong
  3. Control design: Implement controls proportional to risk
  4. Testing: Verify controls work through red-team exercises
  5. Monitoring: Track agent behavior in production
  6. Incident response: Have clear processes for when things go wrong
