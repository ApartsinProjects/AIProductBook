8.3 AI-Enhanced Roadmapping

Objective: Handle uncertainty in AI products through adapted research, prioritization, and data strategies.

Prerequisites: Completion of Section 8.2 recommended.
Big Picture: Planning for AI Uncertainty

AI product roadmaps must account for external dependencies that traditional software does not have: model release cycles, data collection timelines, and evaluation infrastructure requirements. Building model-release-ready infrastructure and starting data collection early are essential practices. The key insight is that AI roadmaps should include explicit uncertainty buffers and always have features that do not depend on external model capabilities.

Fun Note: The Roadmap to Nowhere

The phrase "roadmap" originally referred to maps used for roads. In 1903, the first road maps were handed out at gas stations to encourage car travel. AI roadmaps serve a similar purpose: encouraging investment in a journey whose destination keeps changing.

8.3 AI-Enhanced Roadmapping

Roadmapping for AI products introduces dependencies that do not exist in traditional software development. Model release cycles from providers like OpenAI, Anthropic, and Google can dramatically change what's possible, making long-term planning uncertain. Your roadmap must account for external model availability while building the infrastructure that can take advantage of improvements when they arrive.

Model Release Cycle Dependencies

The AI model landscape evolves rapidly. New model releases frequently introduce capabilities that enable features previously impossible or make existing features significantly cheaper to run. This creates both opportunity and risk for product roadmaps. The key is building flexibility into your roadmap while planning for known model capability trajectories.

Case Study: OpenAI o1 Release Impact

When OpenAI released the o1 reasoning model in late 2024, products requiring complex multi-step reasoning suddenly became viable. Several AI product companies had been planning "reasoning" features for 2025 but were able to ship within weeks of the model release. The key was having the infrastructure ready: they had built the user-facing features but were waiting for a model that could handle the reasoning requirements.

Why this matters: The companies that had prepared infrastructure in advance could capitalize immediately on the model release. Those that had not started would be months behind. This pattern repeats with every major model release.

Skill: Building Model-Release-Ready Infrastructure

When planning AI features, identify which features require specific model capabilities (reasoning, vision, function calling, low latency) and which work with current models. Build the infrastructure for all of them, but mark features requiring next-generation models as "pending model capability." This way, when new models release, you can activate those features quickly.

Example roadmap approach:

  • Q1: Build RAG infrastructure, ship search features (works with current models)
  • Q2: Build agentic workflow infrastructure, plan reasoning features
  • Q3: When reasoning model releases, activate agentic features
  • Q4: Optimize for cost as new efficient models release

Data Collection as a Parallel Track

Data collection is often the longest lead item in AI product development, frequently determining feature timelines more than engineering does. If your AI product requires domain-specific training data, evaluation data, or retrieval knowledge, collecting and preparing that data can take months. Build data collection into your roadmap as a parallel track starting early, independent of engineering progress.

Case Study: Healthcare AI Data Challenges

A health-tech company planned a clinical decision support AI for release in Q2. Engineering completed the AI system in March, but the feature did not ship until August. The delay: they needed physician-annotated clinical scenarios for evaluation, and recruiting, training, and managing annotators took five months. The AI system was technically complete but could not be validated for clinical safety without the evaluation dataset.

Key lesson: Data timelines are not elastic. Start data collection at the beginning of the product planning process, not when engineering is complete.

Skill: Data Dependency Mapping

For each AI feature on your roadmap, identify:

  1. Training data: What data is needed to build or fine-tune the model?
  2. Evaluation data: What golden dataset is needed to validate quality?
  3. Knowledge data: What documents/content must be indexed for RAG?
  4. Feedback data: How will you collect user signals to improve?

For each data type, estimate collection time and work backwards from your target launch date.

Evaluation Infrastructure as Prerequisite

Evaluation infrastructure is not an afterthought for AI products; it is a prerequisite for shipping. Without the ability to measure quality, you cannot reliably ship. Build eval infrastructure early in development and use it continuously to track quality as you iterate. This is particularly important for features that will improve over time: you need baseline measurements to know whether you are actually improving.

Skill: Building Eval Infrastructure Early

The eval infrastructure should include:

  1. Automated evaluation pipeline: Run golden dataset tests on every code change
  2. Quality dashboards: Track metrics over time, not just point-in-time scores
  3. Human evaluation workflow: Systematic process for human raters to assess outputs
  4. A/B testing framework: Infrastructure to compare model versions or prompt variations

Invest in eval infrastructure proportionally to the stakes of your AI feature. High-stakes AI (healthcare, legal, financial) requires more sophisticated evaluation than low-stakes AI (entertainment, productivity).

Case Study: Continuous Eval Saves Launch

A B2B SaaS company built continuous evaluation into their AI email assistant from day one. At launch, automated eval caught a regression: the model had started generating 15% more overly promotional language after a prompt change. The regression was caught before any customers saw it. Without continuous eval, the issue would have affected thousands of users before being noticed through support tickets.

Why this matters: AI quality can degrade silently. Unlike traditional software bugs that cause errors, quality degradation often produces outputs that look acceptable but are systematically worse.

Handling Roadmap Uncertainty

AI product roadmaps should include explicit uncertainty buffers and fallback plans. Build your roadmap with three horizons: features that work with current models, features requiring announced model capabilities, and features requiring unannounced capabilities. This分层 approach ensures progress regardless of model release timing.

The AI Roadmap Rule

Always have at least one feature that does not depend on external model capabilities. This ensures your roadmap progresses even if model providers delay releases.
